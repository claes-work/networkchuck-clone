# Virtual Machines Pt. 2 (Proxmox install w/ Kali Linux)

## Summary

Part 2 of Chuck's virtual machine series, stepping up from a type 2 hypervisor
(VirtualBox, covered in Part 1) to a **type 1 hypervisor** using the free,
open-source **Proxmox VE**. The premise: repurpose an old, dusty laptop or
desktop by wiping its OS entirely and installing Proxmox directly on the bare
metal, turning it into a dedicated "ultimate nerd IT/hacking lab" powerhouse for
running many larger VMs and containers.

The video is a hands-on, start-to-finish tutorial: motivation (why type 1), the
required hardware, prepping a bootable USB, installing Proxmox, post-install
storage maintenance, uploading ISOs, spinning up VMs (Kali, Windows Server 2019,
Ubuntu) and an LXC container, and finally tweaking a laptop so it keeps running
with the lid closed. Chuck frames the whole thing as a résumé-building skill:
learning Proxmox transfers directly to VMware ESXi, which "runs the world" in
corporate IT.

## Key claims (dated 2021-04-27)

Concept — type 1 vs type 2 hypervisors:
- A type 2 hypervisor (e.g. VirtualBox) is an app that runs alongside a host OS
  (Windows/Mac/Linux) and shares that OS's resources, limiting how many and how
  large your VMs can be. (paraphrase)
- A type 1 hypervisor runs directly on the hardware with nothing between it and
  the metal; the host OS is wiped and replaced. Its lightweight OS exists only
  to give VMs the maximum share of CPU/RAM/storage. (paraphrase)
- Type 1 is what businesses actually run in production IT infrastructure — you
  won't find type 2 running real corporate workloads. Learning it is a career /
  résumé builder. (paraphrase)
- Chuck used VMware ESXi in the field, not Proxmox; he tried ESXi for this video
  but it "does not play well with consumer hardware," so he switched to Proxmox,
  which is hardware-compatible, does everything, and is completely free with no
  signup. He judges Proxmox and ESXi similar enough that learning one lets you
  fumble through the other. (paraphrase, self-reported)
- Proxmox does not only run VMs — it also natively runs containers, specifically
  LXC (Linux containers), not Docker. (paraphrase)

Setup — what you need:
- A computer (anything from an old laptop to a decked-out server; Proxmox's real
  hardware requirements are low), a USB flash drive (8 GB minimum), an ethernet
  cable and a switch (Proxmox will not use Wi-Fi — wired only), and coffee.
  (paraphrase)

Setup — install steps (in order):
- Download the Proxmox VE ISO from proxmox.com → Downloads → ISO Images (version
  shown was 6.3). (paraphrase)
- Write the ISO to the USB with Rufus on Windows (Etcher on Mac/Linux). In Rufus,
  select the flash drive, select the ISO, click Start, and choose **DD mode, not
  ISO mode** — ISO mode does not work. (paraphrase)
- Boot the target PC from the USB, changing BIOS boot order if needed (Delete/
  F8/F11/F12 depending on the machine). (paraphrase)
- In the Proxmox installer: plug in the ethernet cable (it needs an IP), accept
  the EULA, choose the target disk, set locale, set the root password, and set
  hostname/FQDN + static IP. (paraphrase)
- After install, remove the USB and manage everything from the web UI at
  `https://<static-ip>:8006`; log in as user **root** with the install password.
  A "No valid subscription" popup is expected and harmless — Proxmox is free and
  open source; the subscription is optional paid support. (paraphrase)

Setup — post-install storage fix (single-disk installs only):
- By default a single-disk install splits storage into `local` (usable for VMs
  but small) and `local-lvm` (large but not usable the way he wants). Fix: in
  Datacenter → Storage, remove `local-lvm`, then in the node's Shell run three
  commands: `lvremove /dev/pve/data`, `lvresize -l +100%FREE /dev/pve/root`, and
  `resize2fs /dev/mapper/pve-root`. This reclaims the whole drive (his 500 GB
  laptop went to ~460 GB usable). Do this before deploying any VMs/containers.
  (paraphrase)

Setup — creating VMs and containers:
- Upload OS ISOs via local storage → ISO Images → Upload (Kali, Ubuntu, Windows
  Server 2019, etc.). To allow VMs/containers on the `local` directory you must
  edit Datacenter → Storage → local and enable the required content types (Disk
  image for VMs, Container for LXC). (paraphrase)
- Create a VM with "Create VM": set VMID/name, pick the ISO, allocate disk (he
  gave Kali 50 GB), CPU cores, RAM (bumped to 4096 MB), and network (bridged to
  the home network / router DHCP by default). Switching between running VMs in
  the web UI and opening each VM's Console is fast and easy. (paraphrase)
- LXC containers spin up in seconds: download a CT template (local → CT Templates
  → Templates), then "Create CT" with name, password, template, storage, and
  network. (paraphrase)
- Windows Server can be downloaded free as an evaluation; Chuck notes his eval
  was valid for 180 days. (paraphrase, self-reported)

Setup — keep a laptop server running with the lid closed:
- Edit `/etc/systemd/logind.conf` (with nano): uncomment `HandleLidSwitch` and
  set it to `ignore`, and set `HandleLidSwitchDocked` to `ignore`; then restart
  the service with `systemctl restart systemd-logind.service`. (paraphrase)
- To let the screen sleep after closing the lid, edit `/etc/default/grub`, add
  `consoleblank=300` inside `GRUB_CMDLINE_LINUX` (screen sleeps after 300 s / 5
  min), then run `update-grub`. (paraphrase)
