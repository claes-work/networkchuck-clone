# your home automation SUCKS!!

## Summary

Chuck walks through setting up [[../topics/homelab-selfhosting/home-assistant]] — self-hosted, local home-automation software — to replace the cloud-based Amazon Alexa he had been relying on. The framing gag is "Operation Toilet": a multi-step automation that punishes his household when the guest-bathroom toilet lid is left up and the door left open (his youngest daughter Jenny throws a roll of toilet paper into the toilet). When no one is detected for a moment and either the lid is up or the door is open, the automation turns every light red and flashing, enables a firewall rule to cut internet for everyone but him, and plays a prerecorded "ransom" video on the house TVs on loop until the lid and door are closed.

Underneath the gag it is a beginner-to-intermediate Home Assistant tutorial. Chuck argues the core reason to run Home Assistant is that it is **local and private** — unlike Alexa, which is cloud-based, so "Mr. Bezos" controls the home and it stops working when the internet is out. He covers two install methods (Raspberry Pi 5 imaged via Raspberry Pi Imager, and a VM on his Proxmox cluster), accessing it at `homeassistant.local:8123` via mDNS, and the core Home Assistant concepts: integrations → devices → entities → areas. He integrates Philips Hue (via the Hue hub, auto-discovered by mDNS/SSDP), his UniFi network (exposing firewall rules and per-port switch stats as controllable entities), a ZigBee adapter (ZHA integration, local low-power mesh), the Home Assistant mobile app (turning his phone into a sensor with location/battery entities), and Plex (to play the ransom video). He then builds the automations from scratch (triggers, conditions, and/or building blocks, helpers/input booleans as toggles, parallel + repeat + wait-for-trigger blocks, YAML editing). Remote access from anywhere is handled by the sponsor, Twingate, deployed as a Docker connector via the Advanced SSH & Web Terminal add-on, with per-resource fine-grained access instead of full-network VPN exposure. This is billed as the first in a home-automation series.

## Key claims (paraphrase unless quoted; dated 2024-10-25)

Setup / how-to:
- Home Assistant runs on nearly anything, including credit-card-sized computers (Raspberry Pi 3/4/5); higher-end Pi gives a better experience. Chuck uses a Pi 5.
- Install on the Pi by writing the Home Assistant OS image to an SD card with the Raspberry Pi Imager (Choose Device → Other specific-purpose OS → Home assistants and home automation → Home Assistant).
- Ethernet is recommended over Wi-Fi; an external SSD/NVMe drive is recommended for longevity because SD cards can die.
- He also installs Home Assistant as a VM on his Proxmox cluster (running on 45Drives hardware) using the provided hypervisor image — running two instances, one for his studio and one for his house.
- After install, Home Assistant is typically reachable at `homeassistant.local:8123` via mDNS (multicast DNS); the same discovery mechanism also finds IoT devices on the network.
- Core concept hierarchy: **integrations** connect Home Assistant to other systems; **devices** are the physical things (a bulb, a sensor); **entities** are the individual controllable/readable data points on a device; **areas** are rooms.
- Integrate Philips Hue via the Hue **hub** (press the physical button to pair), not each bulb directly; Home Assistant auto-discovered Hue via mDNS and SSDP (Simple Service Discovery Protocol).
- The UniFi integration exposes network features as entities — firewall rules become toggleable entities, and a switch (USW-24-PoE) exposes per-port state, CPU and memory utilization; network clients can be made trackable entities.
- ZigBee is added as an integration (ZHA) via a USB dongle or an Ethernet/PoE ZigBee adapter; it creates a local low-power mesh where devices relay messages, and usually does not touch the internet.
- Thread and Matter exist as alternative IoT protocols but are not covered in this video.
- The Home Assistant mobile app (iOS/Android) turns the phone into a sensor, exposing entities like location, battery level, and charging state — enabling automations such as low-battery alerts and a future "Weasley clock" family-location project.
- Add-ons run extra applications alongside Home Assistant; each add-on consumes additional CPU/RAM, so adding many can bog the system down.
- Automations are built from triggers (state changes on an entity), optional conditions (with and/or building blocks), and actions; helpers (e.g. an input-boolean toggle) can act as an intermediate switch that decouples detection from action.
- Advanced automation building blocks used: parallel, repeat-while, and wait-for-trigger (waiting for a Plex player to go idle) to loop media playback cleanly; automations can be edited in a visual editor or directly in YAML.
- Plex integration lets Home Assistant play a specific media file (by content ID) as an automation action and see who is watching.

Local-control / privacy philosophy:
- The single biggest reason Chuck chose Home Assistant is that it is local and private — it runs in his house and no cloud vendor can reach into it.
- Cloud-based smart homes like Alexa mean the vendor ("Mr. Bezos") controls your home, and when the internet goes out, so does your home automation — which he argues is not actually "smart."
- ZigBee's local, non-internet, low-power mesh is a top priority for him when choosing IoT devices.
- Turning his phone into a sensor is fine precisely because the tracking data stays local to him, not sent out to Amazon.
- Home Assistant on your network is only reachable on that network by design; to reach it remotely he uses Twingate (sponsor) rather than exposing anything to the internet.
- Twingate gives fine-grained, per-resource access (e.g. his wife can reach Home Assistant but not his server) instead of a traditional VPN that grants access to the whole network; it deploys as a Docker connector inside the network and is free for his use case.
