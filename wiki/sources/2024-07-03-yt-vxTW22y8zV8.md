# Linux on Windows......Windows on Linux

## Summary

A hands-on tutorial for WSL 2 (Windows Subsystem for Linux, version 2), which runs full
Linux distros (Ubuntu, Kali, openSUSE Tumbleweed) inside Windows using lightweight
virtual machines. Chuck frames it as "Linux and Windows became one person" — bidirectional
interop where Windows commands run inside Linux and Linux commands run inside Windows
(via `wsl`). He walks the complete lifecycle: prerequisites (BIOS virtualization,
Windows 11 or Windows 10 build 19041+), one-command install (`wsl --install`), managing
multiple distros, GUI apps via WSLg, Docker inside WSL, GPU passthrough, config files
(`/etc/wsl.conf` local and `.wslconfig` global, including mirrored networking mode), and
backup/restore of instances via export/import to tar files.

Despite the "Windows on Linux" title, the video is entirely about running Linux inside
Windows via WSL 2. The "reverse" direction Chuck demonstrates is running Windows
commands/tools inside the Linux environment (not a Windows guest VM). VirtualBox is
mentioned only as coexisting with WSL 2 (both now default to Hyper-V), not as a Windows-VM
walkthrough. Docker Desktop uses WSL 2 as its backend. The video includes a sponsor
segment for Incogni (data-broker removal) triggered by demoing the OSINT tool SpiderFoot.

## Key claims (dated 2024-07-03; paraphrase)

- WSL 2 requires a 64-bit processor (ARM supported), a minimum of 4 GB RAM (16 GB+ for
  best experience), and virtualization enabled in BIOS — Intel VT-x or AMD-V (AMD-V) —
  because WSL 2 uses lightweight virtual machines.
- Host OS support: best on Windows 11; Windows 10 supported at build 19041 or higher
  (check via `winver`).
- Install WSL with a single command run as admin in Windows Terminal: `wsl --install`.
  It enables the Virtual Machine Platform service, installs WSL, and installs Ubuntu as
  the default distro; requires a reboot, then prompts for a Linux username/password.
- After install, launching Terminal defaults to PowerShell; re-enter a distro with
  `wsl -d Ubuntu`. Exit a distro with `exit` or by closing the window.
- List installable online distros with `wsl --list --online`; install one with
  `wsl --install <distro>` (Chuck installs openSUSE Tumbleweed this way). Distros can also
  be installed from the Microsoft Store (Chuck installs Kali Linux via the Store because
  the command-line install gave him problems).
- Kali installs minimal; run `kali-tweaks` → Metapackages to install Kali's "top 10" tools.
- Run Windows commands inside Linux by appending/using the `.exe` name: `ipconfig.exe`,
  `ping.exe`, `notepad.exe`, and `explorer.exe .` (the latter opens the current Linux
  directory in Windows Explorer).
- Run Linux commands inside PowerShell by invoking `wsl`: e.g. `ipconfig | wsl grep "10."`
  or `wsl cat <file>`. `grep` filters by search.
- WSL management commands: `wsl --list --verbose` (see running/stopped state and default
  distro, marked with `*`); `wsl --shutdown` (stop all); `wsl --terminate <distro>` (stop
  one); `wsl --set-default <distro>`; `wsl --status`; `wsl --unregister <distro>` (delete
  a distro).
- File interop: from Linux, access the Windows drive at `/mnt/c`. From Windows, access
  Linux files via the "Linux" (penguin) location saved by default in the Windows 11
  sidebar/home bar, or open the current Linux dir with `explorer.exe .`.
- WSLg lets Linux GUI apps launch and render as Windows GUI windows — e.g. `sudo wireshark`
  or launching Google Chrome from Ubuntu. Works out of the box for any GUI app.
- Kali offers `kali-win-kex` (Win-KeX): install with `sudo apt install kali-win-kex -y`,
  run with `kex` for a full desktop session (set a password, "no" for view-only); F8 for
  options, `kex --stop` to stop.
- VS Code integration: type `code <file>` from the Linux terminal to open the Windows-
  installed VS Code; the WSL extension connects VS Code to the WSL distro (shown bottom-
  left), giving terminals, file access, and autocomplete against Linux.
- Docker inside WSL: install via `sudo apt install docker.io`, or install Docker Desktop,
  which uses WSL 2 as its backend (config option "use WSL 2 instead of Hyper-V"). VS Code's
  Docker extension can view/attach to containers running in WSL.
- VirtualBox now coexists with WSL 2 because it can default to using Hyper-V (which WSL 2
  enables), so the historical conflict between WSL and VirtualBox no longer forces a choice.
- GPU access: Linux under WSL can see the host GPU — `nvidia-smi` shows Chuck's RTX 3080
  from inside WSL. Chuck notes doing GPU passthrough with a traditional VM (VirtualBox/
  VMware) is difficult if possible at all.
- Config files: per-distro local config at `/etc/wsl.conf` (systemd is supported and set
  to `true` by default; also auto-mount, hostname/network, interop, and default-user
  settings). Global config at `%UserProfile%\.wslconfig` applies to all instances and is
  stored on Windows (not created by default).
- Networking: default mode is NAT (WSL instances get a separate network, e.g. 172.x). Set
  `networkingMode=mirrored` under a `[wsl2]` section in `.wslconfig` so WSL shares the
  Windows host IP and is reachable on the LAN. Note the documented "8-second rule": wait
  ~8 seconds after `wsl --shutdown` before changes take effect.
- Backup/restore: export an instance to a tar file with `wsl --export <distro> <file>.tar`;
  restore/clone on another machine with `wsl --import <name> <install-path> <file>.tar`.
  Imported instances default to the root user; connect as a specific user with
  `wsl -d <name> -u <user>`.
- OSINT demo: SpiderFoot (`spiderfoot -l 127.0.0.1:5001`) runs a web server inside WSL for
  open-source-intelligence scans; Chuck scans his own name and finds a hacked email tied to
  his father (also named Chuck Keith).
