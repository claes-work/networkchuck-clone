# Google is forcing you to use their proxy (build your own instead)

## Summary

Chuck reacts to Google Chrome's proposed "IP Protection" feature — a Google-run
"privacy proxy" that would sit between users and websites to hide their IP address —
and uses it as motivation to teach viewers to self-host their own proxy instead. He
first explains what a proxy is (a server in the middle that swaps your IP for its own),
how it differs from a VPN (a proxy forwards traffic without encrypting it, so it is
faster and lighter to host; a VPN encrypts over a secure tunnel), and how a proxy also
lets you appear to be located anywhere in the world.

He then critiques Google's approach: putting Google in the middle of everyone's traffic
resembles a man-in-the-middle position and creates a single point of failure (one hacked
proxy = mass data exposure; DDoS against the proxy could destabilize the internet). He
walks through Google's stated mitigations (OAuth tokens tied to a Google account, blinded
signatures, token rate-limiting/expiry, and a proposed two-hop design where a non-Google
CDN like Cloudflare or Akamai runs the second hop — reminiscent of Tor). The rollout is a
phased, opt-in test starting between Chrome M119 and M125, US-only, Google-domains-only
at first.

The core message: rather than let Google own your privacy, own it yourself. He
demonstrates two ways to run your own proxy on a cheap ($5/month) Linux cloud server
(Linode by Akamai): (1) a SOCKS proxy over an SSH tunnel — zero extra software, just
OpenSSH; and (2) Shadowsocks, a SOCKS proxy that also encrypts traffic, combining VPN
and proxy benefits.

## Key claims (paraphrase; dated 2023-11-22)

Concept / how proxies work:
- A proxy is a server that sits between you and a website; the website sees the proxy's
  IP address instead of yours, hiding your identity.
- A proxy can also make you appear to be located anywhere — put the proxy in France and
  websites treat you as being in France (bypassing geoblocked content).
- A proxy differs from a VPN: a VPN encrypts your traffic over a secure tunnel, while a
  proxy just forwards traffic without encrypting it. Because it skips encryption, a proxy
  is typically faster and much less resource-intensive to host.

Privacy motivation / critique of Google:
- Websites can see your IP address, and (per Google's own document) an IP makes it
  feasible to associate what you're doing with where you've been, building a unique,
  persistent profile to track you across the web.
- Unlike cookies, IP-address tracking is covert — you can't just opt out; it happens
  automatically.
- Google Chrome's "IP Protection" feature routes traffic through a Google-owned "privacy
  proxy," which structurally puts Google in a man-in-the-middle position with the
  potential to see everything (Chuck notes Google already sees most of our traffic anyway).
- Google claims the proxy will not be privy to the contents of the traffic ("trust us").
- A Google-run proxy is a single point of failure: hacking one proxy could expose many
  users' data, and a DDoS against it could disrupt internet stability and existing
  online-attack defenses.
- Google's mitigations: authentication via a Google account with OAuth tokens, blinded
  signatures so traffic can't be tied to identity, token rate-limiting and expiry against
  DDoS, and a considered two-hop design (Google first hop + a non-Google CDN second hop,
  like Cloudflare or Akamai) so neither hop sees both client IP and destination —
  similar to Tor's onion relays.
- Because Google must comply with local content-localization laws, its proxy assigns IPs
  representing the user's coarse/approximate location including country — which Chuck
  frames as all the more reason to run your own.
- Rollout is phased and opt-in: phase zero is a single Google proxy handling only
  Google-owned domains, US-based IPs only, small percentage auto-enrolled, requiring the
  user to be logged into Chrome to generate the auth token; expected between Chrome M119
  and M125 (video recorded on Chrome 118).
- Chuck's stance: he's "on the fence" and somewhat glad Google is helping most people
  with privacy, but he'd rather own his own privacy than have Google do it for him.

Self-host setup:
- All you need for either method is a small Linux server; Chuck spins up a $5/month
  shared-CPU Ubuntu instance on Linode by Akamai, choosing the region (Osaka, Japan) to
  control apparent location.
- Method 1 — SSH SOCKS tunnel: with OpenSSH already installed, run
  `ssh -D 1337 -N -C root@<proxy-server-ip>` to open a compressed SOCKS proxy tunnel
  (`-D` opens the SOCKS proxy on a port, `-N` forwards ports without executing remote
  commands, `-C` compresses to save bandwidth), then point the client at it.
- Configure the client (Chuck recommends Firefox on Windows) with manual proxy: SOCKS
  host `localhost`, port `1337` — the browser then routes through the tunnel to the
  remote server, so websites see the server's location (Japan) instead of Dallas, Texas.
  This also works globally on Ubuntu via system proxy settings.
- Method 2 — Shadowsocks: a SOCKS proxy that also encrypts traffic, combining VPN and
  proxy. Deployed one-click from the Linode Marketplace (Chuck picks Mumbai), with a
  Shadowsocks client installable on Windows, Mac, Linux, and phones. Config lives in
  `/etc/shadowsocks-libev/config.json` on the server (default port 8000, encryption
  method AES-256-GCM); enter server IP, port, password, and encryption method in the
  client to route (and encrypt) traffic.
