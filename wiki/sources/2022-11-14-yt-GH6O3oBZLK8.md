# subdomain takeover (stealing websites)

## Summary

Chuck demonstrates a **subdomain takeover** attack using a fictional target,
`hackwellindustries.com`. The setup: subdomains (e.g. `code.hackwellindustries.com`,
`azure.hackwellindustries.com`) point via CNAME records to third-party services like
GitHub Pages or Microsoft Azure. When the org's admin ("Bernard Hackle") deletes the
underlying cloud resource (the GitHub account, the Azure resource) but leaves the DNS
CNAME entry in place, the DNS record becomes "dangling" — it still points at a
now-unclaimed name on the third-party service. An attacker can then simply re-register
that freed name (create a GitHub account / Azure resource with the same identifier),
and the victim's real subdomain now resolves to attacker-controlled content.

Chuck walks the full workflow: (1) enumerate subdomains with **amass** (run via Docker),
(2) test them for takeover vulnerability with a tool called **takeover** (checks against
many services), (3) confirm the target with **dig** to read the CNAME, then (4) claim
the freed name. For GitHub Pages he creates a `username.github.io` repo with a `CNAME`
file naming the victim domain; for Azure he spins up a VM in the matching region and
claims the freed DNS label. He notes GitHub reserves a deleted username for 90 days
(his live demo hit this), but Azure and services like S3 buckets have no such hold.

The attack is framed as **untraceable** and dangerous because it enables convincing
phishing on a real, legitimate domain (credential theft). It is presented explicitly for
**educational / bug-bounty purposes**: the defensive takeaway is to clean up stale DNS.
(A mid-video sponsor segment for Contrast Security "CodeSec" code scanning is
promotional, not persona content.)

## Key claims (dated — 2022-11-14)

**Concept / attack:**
- Subdomains often point via CNAME to third-party services (GitHub, Azure), and this is
  normal until the underlying resource is deleted. (paraphrase)
- The vulnerability is a **dangling DNS record**: deleting the cloud resource (GitHub
  account, Azure resource) without also deleting the DNS entry leaves a CNAME pointing at
  an unclaimed name. (paraphrase)
- An attacker re-registers that freed name on the third-party service and thereby
  controls what the victim's subdomain serves. (paraphrase)
- The attack is untraceable — no alarms fire, no scanning catches it, and the org
  typically won't notice. (paraphrase, Chuck's framing)
- It enables high-quality phishing: victims see the real domain, so credentials can be
  stolen. (paraphrase)
- Finding vulnerable subdomains: enumerate with **amass** (run in a Docker container),
  then test candidates with the **takeover** tool, which checks against many services;
  confirm the CNAME target with **dig**. (paraphrase)
- Taking over GitHub Pages: create a public repo named `username.github.io`, add HTML,
  and add a `CNAME` file containing the victim's domain; GitHub's Pages DNS check then
  serves attacker content on the victim subdomain. (paraphrase)
- GitHub keeps a deleted username reserved (unavailable) for **90 days**; after that it
  is claimable by anyone. Azure and other services (e.g. S3 buckets, storage services)
  have no such waiting period. (paraphrase)
- Taking over Azure: create a VM in the same region (central US in the demo) and claim
  the freed DNS label, immediately controlling the victim subdomain. (paraphrase)

**Defense:**
- Protection is to **clean up stale/dangling DNS entries** — audit your DNS and confirm
  every entry is still used by you, not left pointing at a released resource. ("Don't be
  lazy.") (paraphrase)
- Even perfect DNS is not enough if the app/code itself is insecure — secure code still
  matters. (paraphrase)

**Ethics / framing:**
- Presented for educational purposes only; the goals are (a) show the vulnerability is
  real so viewers check their DNS, and (b) show bug-bounty hunters a legitimate skill to
  scan for. (paraphrase)
