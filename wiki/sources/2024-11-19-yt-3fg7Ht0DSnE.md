# Cloning my Voice Into an AI Assistant

## Summary

A continuation of Chuck's local AI voice-assistant build. In the previous video he built "Terry," a completely local AI voice assistant (running on his dual-RTX-4090 AI server) that replaced Alexa, but he couldn't give it a custom voice. This video solves that: cloning a voice locally, free, and open-source instead of paying for a cloud service like ElevenLabs. The pipeline uses **Piper TTS** (referred to in the captions as "Hyper TTS") to train/fine-tune a text-to-speech voice model exported to the universal **ONNX** format, which can be dropped into any TTS program or into Home Assistant's Piper integration.

Chuck walks through two ways to gather training data: **Piper Recording Studio** (a web UI on port 8000 for recording your own voice by reading prompts) or scraping audio from YouTube videos with **yt-dlp**, cleaning it in **Audacity**, splitting into <15-second clips, removing silence with **ffmpeg**, and auto-transcribing with **Whisper** into a `metadata.csv` + `wav/` dataset. Training is done by fine-tuning an existing English (lessac) checkpoint from Hugging Face rather than starting from scratch. He demonstrates across three hardware setups: an RTX 3080 laptop in WSL, his dual-4090 server (Terry), and a cloud AWS EC2 G5.12xlarge instance (~$5/hr, four GPUs).

Results were mixed: his own voice cloned from fast-talking YouTube clips sounded terrible, but re-doing it with ~400 Piper Recording Studio utterances gave a usable result. Cloning his editor Mike's voice and pastor Jami's / David Bombal's voices came out much better. The takeaway: clip quality (clean, no music/typing), quantity, diversity, and epoch count determine output quality. He ends by loading the ONNX model into Home Assistant / Terry so the assistant speaks in the cloned voice, giving Terry a "sinister/unhinged" persona via the system prompt.

## Key claims

All dated 2024-11-19 (paraphrase unless quoted).

Build / tooling:
- The whole voice-cloning stack can be done **locally, open-source, and free**; Chuck avoids cloud tools like ElevenLabs because they are expensive and cloud-based (not private).
- Core software is **Piper TTS** (the captions repeatedly garble it as "Hyper TTS"); the trained voice is exported to the **ONNX** universal TTS format so it can be used anywhere.
- Requirements: any computer (Mac, Linux, Windows). A GPU dramatically speeds training; a CPU works but is far slower.
- Demonstrated on three GPU setups: RTX 3080 laptop (in WSL/Ubuntu 22.04), the dual-RTX-4090 server "Terry," and a cloud AWS EC2 **G5.12xlarge** instance with four GPUs (~$5/hour).
- Step 1 is clean data. Two data sources: **Piper Recording Studio** (records your own voice via a web UI on `localhost:8000`) or pulling audio from existing content.
- To scrape a voice from YouTube, download audio with **yt-dlp** (`-x --audio-format wav`); pick clean clips with no background music or interruptions.
- Data cleaning chain: **Audacity** to trim music and export mono / 22050 Hz wav; **ffmpeg** to remove silence; a bash for-loop to split files into clips no longer than **15 seconds** (Piper's limit).
- Transcription is done with local **Whisper** (`pip install whisper`), producing a `metadata.csv` mapping each wav file to its transcription.
- Dataset format Piper expects: a `wav/` folder of audio plus a `metadata.csv`.
- Training uses **fine-tuning from an existing checkpoint** (an English US "lessac" medium-quality model downloaded from Hugging Face) rather than training from scratch — described via a relay-race baton analogy.
- Piper depends on older libraries; Chuck had to pin specific versions (pip 23.3.1, numpy 1.24.4, a specific torchmetrics, a PyTorch version) and edit the requirements file. He ran `pip3 install -e .` and `./build_monotonic_align.sh`, then `piper_train.preprocess`.
- **CUDA 11.7 (Piper's required version) has a bug that breaks on RTX 4090s**; a community fix (credited to user "LPSCR") — editing the requirements file plus installing a specific PyTorch version — was needed to train on the 4090s.
- Training knobs: `--batch-size` depends on GPU VRAM (32 for the 3080; lower to 16/8/4/2 on errors); `--max_epochs` (he uses 6000, resuming from epoch 2164); `--precision 16` (mixed precision) claimed to possibly help on 4090s; multi-GPU via a `ddp` strategy and `--devices`/GPUs flag; training is resumable from the latest checkpoint in `lightning_logs`.
- Export: `piper_train.export_onnx` converts the final checkpoint to `model.onnx`, plus copy `config.json` to `model.onnx.json`. `pip install piper-tts` then lets you generate speech via `echo "text" | piper -m model.onnx --output_file out.wav`.
- Deployment into the assistant: upload the ONNX files to Home Assistant via a Samba share, use the local **Piper** add-on / **Wyoming** integration, and select the voice under text-to-speech. Terry also runs Piper via a **Docker** container.

Quality caveats:
- His own voice cloned from YouTube clips sounded very bad; he attributes this to talking too fast and to clips that weren't pristine (music, keyboard typing, background noise degrade quality).
- Re-recording ~400 utterances in Piper Recording Studio produced a usable ("not perfect, but good enough for a voice assistant") result.
- Output quality is driven by clip quality/cleanliness, clip quantity, diversity of what's said, and number of epochs trained.

## Notable verbatim quotes

Voice-cloning / AI-ethics caveats (spoken as an on-screen disclaimer):

> "Now, disclaimer, you can't just clone a voice and use it for whatever you want."

> "I'm going to show you how to clone my voice from my YouTube videos. I'll also show you how to clone your voice, so I'm getting you permission to clone my voice, but you cannot use my voice for any commercial purposes to produce any content and don't distribute it anywhere, and that goes for any voice you make, but I would love to be your voice assistant."

> "Now, asterisk, this is not legal advice. Just keep everything local, and if you have doubts, consult with a lawyer."

On why local/open:

> "There is software out there that makes it easy to clone a voice like 11 labs, but I hate that because it's expensive. Also, it's in the cloud. So I want to clone a voice with local and open source software, and I want it to be completely free."

> "This is why I love doing this local and open. You get more power and it's private."

> "It just hits different to use something local on your own hardware, doesn't it? Way more fun."

On the build being hard:

> "This video almost killed me. I spent hours and days trying to figure this out."

> "While Piper is incredible, it's working with some older libraries that don't work with modern stuff."

On data quality:

> "The first thing you have to do with any AI project is make sure you have good clean data."

> "It comes down to the quality of the clips you have, the number of the clips you have and the diverse number of things they might be saying. Also, the number of epochs or the number of times you train will impact the quality."

> "Whenever you're training a voice, the quality of the voice, background, music, keyboard typing can all interfere with the quality at the end."

On his own failed clone:

> "That was the worst thing I've ever heard... that did not work for me for some reason."

> "I think it's because I talked too fast."
