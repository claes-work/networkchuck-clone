# Your Remote Desktop SUCKS!! Try this instead (FREE + Open Source)

## Summary

Chuck walks through the state of remote-desktop / remote-access tooling and lands on **RustDesk** as his favorite free, open-source, self-hostable solution. He frames it against native and commercial options: Windows' native RDP/RDC (called out for requiring Windows Pro on the target machine, ~$200, and only working same-network), Mac's Microsoft Remote Desktop app, Linux `xrdp`, and the VNC protocol (which he finds painful to install, especially on Linux). He gives honorable mentions to browser-based remote desktop (Apache Guacamole and his favorite, Kasm VNC / Kasm Workspaces) as great for lightweight web/terminal access but not native clients. He rejects TeamViewer (nag screens, and it routes through TeamViewer's servers — "they're in control") and AnyDesk (self-hosting its server is a paid Pro feature), choosing RustDesk because it's written in Rust, open source, has GPU-accelerated native clients for every OS including phone, and — the deciding factor — lets you host your own server locally for free.

The tutorial covers two usage modes. First, **direct IP-to-IP local access**: install the RustDesk client, enable "direct IP access" (default port 21118), set a permanent password, and connect over the LAN with no cloud/server involved (shown as "direct and unencrypted" on a local network). Second, **self-hosting a RustDesk server** (a signaling/ID server `hbbs` + relay server `hbbr`) via Docker Compose on a cloud VM (he uses a $5/month 1 GB Ubuntu 24.04 Linode). The server acts as a rendezvous/relay: given a device **ID** (like TeamViewer), it does NAT hole-punching for a direct connection or relays traffic when a direct path isn't possible. He shows grabbing the server's public key (`id_ed25519.pub`) and IP and configuring each client's ID/relay server so clients stop using RustDesk's public default servers — a privacy win.

Throughout he ties this to remote access via the sponsor **Twingate** (zero-trust network access; free for homelabs and small businesses under ~5 users), which he pairs with RustDesk for secure local connectivity from anywhere. Context: NetworkChuck Studios is moving all video editors from PC to Mac, and Windows can't easily remote into Mac natively — a real problem RustDesk solves.

## Key claims

_All paraphrased unless quoted. Dated 2025-02-04 (video publication)._

- (2025-02-04) Chuck's chosen remote-desktop solution is **RustDesk** — free, open source, self-hostable, cross-platform (Windows/Mac/Linux + phone), with GPU-accelerated native clients. The name comes from being written in the Rust programming language.
- (2025-02-04) His two decisive requirements: it must be **open source** and let you **host your own server locally for free** — which RustDesk does but AnyDesk gates behind a paid Pro tier.
- (2025-02-04) Windows native RDP/RDC's big limitations: the target machine must run **Windows Pro** (~$200; "kind of stupid for something that should be guaranteed built into your system"), and it only works when both machines are on the same network.
- (2025-02-04) Windows cannot easily/natively remote into Mac; Mac can remote into Windows (via Microsoft Remote Desktop, target still needs Pro). This matters because NetworkChuck Studios is moving all editors from PC to Mac.
- (2025-02-04) Linux can accept RDP connections via `xrdp`; the VNC protocol works across OSes but is a pain to install (especially on Linux) and can have troublesome copy-paste.
- (2025-02-04) Honorable mentions — **browser-based** remote desktop: Apache Guacamole and his favorite **Kasm** (Kasm VNC / Kasm Workspaces). Great for basic web browsing / terminal work; not native clients, so not ideal for heavy work like video editing.
- (2025-02-04) **TeamViewer** is rejected: nag screens pushing a paid license, and it routes through TeamViewer's own servers — "they're in control." He uses it as the unofficial IT guy at his church but wants something self-hosted for business.
- (2025-02-04) RustDesk usage mode 1 — **direct IP access** on the LAN: enable it in Security settings, default port **21118**, set a permanent password; no server needed if clients are on the same network (or reachable via Twingate).
- (2025-02-04) RustDesk usage mode 2 — **self-hosted server** via Docker Compose brings up two containers: `hbbs` (signaling/ID server) and `hbbr` (relay server). He deploys on a **$5/month, 1 GB RAM Ubuntu 24.04 Linode** cloud VM; local install works too but may need port forwarding.
- (2025-02-04) The server enables **ID-based connections** (like TeamViewer): it discovers the target by ID and either facilitates NAT/firewall hole-punching for a direct connection or acts as the middleman relay. Connection status shows "direct and encrypted" vs "relayed and encrypted."
- (2025-02-04) To use your own server privately, configure each client's **ID/relay server** with the server IP plus the public key from `data/id_ed25519.pub` — otherwise clients default to RustDesk's public servers, which is a privacy concern.
- (2025-02-04) Philosophy: he prefers self-hosting because "I love hosting things myself" and wants a solution that is "truly free and open source" and doesn't make him "feel guilty every time I used it."
- (2025-02-04) Sponsor **Twingate**: zero-trust network access, free for homelabs and businesses under ~5 users; install a connector (even on a Raspberry Pi) to get secure remote access from anywhere. He pairs it with RustDesk for fine-grained access control (e.g., a "video editors" security group gating access to editors' Macs).
- (2025-02-04) RustDesk recently added a **headless Linux** experience for GUI-less servers; Chuck tested it and says "it's not great" but hopes they improve it. Also notes auto-discovery of other RustDesk hosts on the same subnet, and that address books / login require the paid Pro version.
