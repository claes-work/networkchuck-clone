# you need this FREE CyberSecurity tool

## Summary

Chuck walks through deploying **Wazuh** (repeatedly caption-garbled as "waza"/"wza"/"wsu"; the underlying agent files reveal its OSSEC lineage), a free and open-source SIEM/XDR platform. He frames it as a "SIEM" — Security Information and Event Management — the tool the blue team (defensive side) uses to detect and respond to attacks. He argues it's a no-brainer: free, open source, protects your devices, teaches you a ton about security and hacking, and is resume-worthy ("I run my own SIEM").

Architecture: a Wazuh server collects logs from lightweight **agents** installed on monitored endpoints (Mac, Windows, Linux), giving a single-pane-of-glass view with alerting (email/Slack) and **active response** (e.g., firewall-drop an attacking IP).

He demonstrates two deployment paths, both sponsored/hosted on Linode (Akamai):
1. **Marketplace one-click** deploy of the Wazuh app on a Linode (requires the 4 GB plan — the 2 GB won't run it).
2. **Docker** deploy on a smaller 2 GB Linode (clone the `wazuh-docker` repo, `single-node` dir, generate self-signed certs, `docker-compose up -d`).

He then tours the dashboard and demonstrates its modules hands-on: MITRE ATT&CK mapping, compliance checks (PCI, GDPR, NIST, HIPAA), Security Configuration Assessment (SCA/CIS benchmark — his Ubuntu scored 39%), security events, vulnerability detection (CVEs, disabled by default), File Integrity Monitoring (real-time file + Windows registry monitoring), active response (auto-blocking a brute-force via Hydra), and Slack alert integration via incoming webhook.

## Key claims (dated — the tool + setup)

All dated 2023-07-19 (paraphrase):

- The tool is **Wazuh**, a free and open-source SIEM (formally: Security Information and Event Management); it's what the blue/defensive team uses to stop attackers.
- Architecture is server + agents: one Linux server collects logs from agents deployed to endpoints (Mac, Windows, Linux). Chuck deployed the server in ~5 minutes and pushed agents to all his machines.
- You need only two things: a Linux server/computer and something to monitor. Most Linux flavors supported; he uses Ubuntu on a Linode cloud VM. Can also run on-prem, in Docker, or from a provided OVA on VirtualBox.
- Server minimum requirements: 2 GB RAM / 2 CPU cores; recommended for heavier logging: 4 GB RAM / 8 CPU cores.
- **Marketplace install (Linode):** Create Linode → Marketplace → search Wazuh; supply email (for SSL cert), a limited sudo user, and passwords; pick Ubuntu image and a nearby region. Must select the **4 GB plan** — the install fails on smaller (he tried). Install takes ~6 minutes; monitor progress with `htop` watching the `dpkg` process.
- Retrieve credentials after install: `cat` the `.deployment-secrets.txt` file (found via `ls -al`) for the admin username/password. Log in at `https://<reverse-DNS-name>` (grab RDNS from Linode's Network tab).
- **Docker install:** works on a 2 GB Linode (1 GB fails — containers won't run). `sudo apt update`, install `docker.io` and `docker-compose`, clone the Wazuh Docker git repo, `cd wazuh-docker/single-node`, run the provided compose command to generate self-signed certs, then `docker-compose up -d`. Default login is `admin` / password `SecretPassword`.
- **Adding agents:** dashboard "Add agent" → deploy new agent → pick OS/architecture → enter the server's FQDN (reverse-DNS name or IP) → name it/assign a group → run the single install command it generates. Linux uses a one-liner + `systemctl` enable-as-service commands. Windows uses a PowerShell one-liner run **as administrator**, then `net start wazuhsvc` (Windows 7+; no Windows ARM support).
- **MITRE ATT&CK module** maps techniques the machine may be vulnerable to or actively attacked with.
- **Compliance module** checks against PCI, GDPR, NIST, HIPAA standards.
- **SCA (Security Configuration Assessment)** runs a CIS benchmark and scores config hardening (his Ubuntu: 39%, failed 715 checks) — and teaches, giving each finding's rationale, remediation, verification, and associated MITRE techniques (e.g., disable USB storage, don't install Telnet, enable UFW, disable SSH root login).
- **File Integrity Monitoring (FIM):** monitors files and Windows registry keys typically altered during a hack. Configured in the agent's `ossec.conf` (Windows: `C:\Program Files (x86)\ossec-agent\ossec.conf`) under the `syscheck` section. Adding `realtime="yes" report_changes="yes" check_all="yes"` to a monitored directory gives near-instant (~5s) alerts on file add/modify (reports the changed content via checksum). Registry monitoring isn't real-time but scans on an interval (default every 12h / 43200s — changeable, e.g., to 30s). Malware commonly modifies registry `Run`/`RunOnce` keys for persistence.
- **Vulnerability detection** must be enabled on the server (`vulnerability-detector` → enabled `yes`, restart manager); agents come configured. Then it flags known CVEs per app (e.g., a bad VLC version = high-severity CVE; a server with 170+ critical CVEs).
- **Active response:** auto-executes actions on alerts. Demo used a Hydra brute-force against a public box; adding a `firewall-drop` active-response block keyed to the triggering rule ID (57110) blocks the attacker's IP for a configurable timeout (180 seconds) — after which access returns.
- **Alerting:** supports email and Slack. Slack setup: create a Slack app with an incoming webhook for a dedicated channel, then add the integration block (with the webhook URL, optional severity/rule-ID filter) to the server config under `global`.
