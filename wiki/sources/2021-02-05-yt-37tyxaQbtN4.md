# you need to learn Hybrid-Cloud RIGHT NOW!! // FREE CCNA // EP 10

## Summary

Episode 10 of Chuck Keith's FREE CCNA series introduces cloud and hybrid-cloud
concepts for network engineers, framed with his signature "you need to learn X RIGHT
NOW" hook. Chuck uses his own [[../entities/networkchuck-coffee]] website as a running
example to contrast **on-prem** (on-premises) infrastructure with the **public cloud**.

He explains that every app, service, and website needs infrastructure — servers,
routers, switches, databases, firewalls — which traditionally lives in a private data
center (Chuck shows his own two Dell servers running his house and business). He lays
out why the cloud is attractive: it converts huge up-front capital expenditure (capex)
into pay-as-you-go operational expenditure (opex) — e.g. renting an AWS server for ~10
cents/hour instead of buying a ~$100k server — plus **elasticity** (spinning servers up
and down in a few clicks) and **cloud-native features** like containers and Kubernetes
(the microservices model, contrasted with the old monolithic-app + VM approach).

He then argues the counterpoint: **not everything should go in the cloud.** Compliance
and regulatory rules (e.g. GDPR, a joking CIA/NSA example), large storage
requirements, and low-latency needs are reasons to keep some workloads on-prem. The
answer to "cloud or on-prem?" is **both** — that's **hybrid cloud**.

Chuck then vents two "beefs" with hybrid cloud: (1) on-prem lacks the cool cloud-native
features, and (2) it's a management nightmare — separate skill sets and portals for
on-prem and for each cloud, worsened by **multi-cloud** (companies average ~5 cloud
providers: AWS, Azure, GCP), each with its own portals, services, and certifications.
As the sponsored solution, he presents **Dell Technologies Cloud + VMware Cloud
Foundation**: run the same VMware tools (vCenter, vRealize) you use on-prem across AWS,
Azure, and GCP, do vMotion migrations between on-prem and cloud, and get cloud-native
Kubernetes/containers on-prem via VMware **Tanzu** on a Dell EMC **VxRail**
(hyper-converged infrastructure). The theme: one set of tools to manage everything,
empowering existing VMware/server engineers instead of forcing them to learn every
cloud separately.

## Key claims (dated — the concepts)

All dated 2021-02-05 (paraphrase unless quoted):

- Every app, service, and website relies on infrastructure — servers, routers,
  switches, databases, firewalls.
- **On-prem / on-premises** = infrastructure you own and run in your own data center;
  the traditional model, prized for control.
- The **cloud is essentially renting someone else's servers** instead of buying your
  own.
- Cloud shifts a large up-front **capital expenditure (capex)** — e.g. ~$119k for a
  server, ~$47k for a router — into pay-as-you-go **operational expenditure (opex)**,
  e.g. ~$0.10/hour for an AWS server.
- **Elasticity** is a core cloud advantage: you can spin one server into many (and back
  down) in a few clicks/moments to absorb traffic spikes, versus buying hardware that
  costs money, takes weeks/months, and needs expertise.
- **Microservices** is the modern approach: break one big "monolithic" app into
  smaller services; developers can ship code and update apps faster.
- Instead of deploying apps to VMs on a server (the on-prem way), microservices deploy
  each app inside its own **container**; many containers are deployed and managed with
  **Kubernetes** (a tool to deploy, manage, and network containers together; shortened
  to k8s).
- Containers and Kubernetes are **cloud-native** features/tools — traditionally the
  cloud has been the best place to run them because the tooling is baked in, whereas
  on-prem was not built for it (though Chuck notes this is changing).
- **Not everything should go in the cloud.** Reasons to keep workloads on-prem:
  compliance/regulatory rules (e.g. GDPR), very large storage requirements (expensive
  in the cloud), and low-latency needs (apps that must sit next to end users).
- **Hybrid cloud** = using both cloud and on-prem, putting apps in the cloud when it
  makes sense and keeping them on-prem when it doesn't; most companies do this.
- **Multi-cloud** = using multiple cloud providers at once; companies average around
  **5 cloud providers** because different vendors offer better features or lower cost.
- Each cloud (AWS, Azure, GCP) has its own portal, services, deployment methods, and
  certifications — creating a heavy management and staffing burden across on-prem plus
  each cloud.
- **VMware ESXi** is the type-1 hypervisor used on-prem to create and manage virtual
  machines; **Dell** makes the servers those data centers run on.
- **Dell Technologies Cloud + VMware Cloud Foundation** lets you run the same VMware
  management tools (vCenter, vRealize) on-prem AND inside AWS, Azure, and GCP — one
  toolset to manage everything, treating cloud providers like additional data centers.
- **vMotion** lets you migrate virtual machines between locations; with VMware Cloud
  Foundation you can vMotion VMs from on-prem to the cloud (and reverse) with minimal
  disruption (Chuck's demo dropped only one ping).
- VMware Cloud Foundation is a **software-defined data center (SDDC)** — its core value
  is **automation** of networking and infrastructure.
- **VMware Tanzu** brings Kubernetes/containers to on-prem, so cloud-native workloads
  become native on-prem too — managed the same way across on-prem and clouds.
- **Dell EMC VxRail** is a **hyper-converged infrastructure** appliance with VMware
  Cloud Foundation baked in, ready to run containers/Kubernetes on-prem or in the cloud.
- Benefit of one unified toolset: saves time and money and reduces manual error (many
  different ways of doing things eventually leads to mistakes); it also empowers
  existing on-prem VMware/server engineers rather than forcing them to learn every
  cloud.
