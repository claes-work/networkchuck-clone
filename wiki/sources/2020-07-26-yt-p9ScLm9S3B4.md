# What is a ROUTER? // FREE CCNA // EP 2

## Summary

Episode 2 of Chuck's FREE CCNA course (sponsored by Boson Software), teaching what a
router is and why networks need one. Building on EP 1 (the switch), Chuck uses a Cisco
Packet Tracer lab with a recurring character named "Johnny" (a Raspberry Pi) who wants
to order coffee from `networkchuck.coffee` — a web server that lives on a *different*
network. The switch alone can't get Johnny there; he needs a router.

The core teaching arc: a **switch** operates at Layer 2 (MAC addresses) and only
connects devices on the *same* network; a **router** operates at Layer 3 (IP addresses)
and connects *different* networks. Chuck defines a "network" as a group of IP addresses
(e.g. `10.1.1.0`–`10.1.1.255`), and shows that two devices are on different networks
because they have IP addresses in different ranges — not merely because they hang off
different switches. He demonstrates a physical Cisco 2911 router (and mentions Ubiquiti
and home routers as other form factors) then drops a virtual 2911 into the lab.

Using Packet Tracer's simulation ("bullet time") mode, Chuck walks frame-by-frame
through three scenarios: (1) Johnny pings a device on his own network (Mark) and learns
his MAC via ARP broadcast; (2) Johnny pings the off-network coffee server and instead
ARPs for his **default gateway** `10.1.1.1` (the router) because he knows the target is
off-network; (3) with the router in place, the packet reaches the router, which ARPs
onward for the server, forwards the packet, and returns the reply. He then shows the
full HTTP/DNS flow to actually load the website (including forgetting to configure the
DNS server the first time), and finishes at the router CLI with `show ip route` and a
peek at a real internet router's massive `show bgp ipv4 unicast` table.

## Key claims (dated 2020-07-26 — paraphrase unless quoted)

- A router's job is to **connect networks**; a switch connects devices within one
  network. Routers come in many form factors (Cisco 2911, Ubiquiti, home routers) but
  the function is the same.
- A "network" really means a **group of IP addresses** (a range), not just "devices on
  the same switch." In the lab, `10.1.1.0`–`10.1.1.255` is one network and
  `23.227.38.0`–`23.227.38.255` is a completely separate network.
- Connecting two switches directly just makes "one giant switch" — it does not join two
  separate networks, because the separation is defined by the IP addressing, not the
  cabling.
- A switch is a **Layer 2** device: the only language it speaks is MAC addresses. It
  cannot see IP addresses.
- A router is a **Layer 3** device: it handles IP addresses. Its data unit at Layer 3
  is a **packet**; at Layer 2 the same data is a **frame**.
- **ARP (Address Resolution Protocol)** is how a host discovers the MAC address tied to
  a known IP address. The ARP request is sent to the Layer 2 **broadcast address**
  (`FFFF.FFFF.FFFF`), so every device on the network hears it; only the owner of the
  target IP replies with its MAC.
- To ping a device on the same network (Mark, `10.1.1.2`), Johnny ARPs for that exact
  target IP. Only Mark responds; Johnny then sends the ping frame directly to Mark's MAC.
- To reach an **off-network** IP (the coffee server `23.227.38.65`), Johnny does **not**
  ARP for the server's IP. He knows it's outside his range, so he ARPs for his
  **default gateway** (`10.1.1.1`) instead. "Gateway" is another word for router.
- A host knows which addresses are local ("in the neighborhood") vs. remote by comparing
  the destination IP against its own network range; remote destinations must be sent to
  the router.
- Without a router configured, the ARP requests for `10.1.1.1` go unanswered and the
  ping fails — the host has nowhere to send off-network traffic.
- Routers need **pre-configuration** to route between networks (unlike switches, which
  largely work out of the box). A router holds a "map" of how to reach networks.
- When Johnny sends his packet, **Layer 3 = directions for the router** (source IP =
  Johnny, destination IP = coffee server) and **Layer 2 = directions for the switch**
  (source MAC = Johnny, destination MAC = the router). The switch delivers the frame to
  the router; the router takes care of the rest.
- The router receives the packet on one interface (e.g. `GigabitEthernet0/0`), must
  build a *new* frame to forward it out the exit interface, and therefore has to ARP for
  the next hop's MAC (the server) before it can send — the outbound frame initially has
  no Layer 2 destination.
- Devices **cache** learned MAC-to-IP mappings so the ARP process doesn't repeat every
  time; cache entries eventually expire and are relearned.
- You browse to a website by **name** (`networkchuck.coffee`), but neither the switch
  (MAC) nor the router (IP) understands names — so the computer must resolve the name to
  an IP first. That resolution is **DNS (Domain Name Service)**. The host must be
  configured with a DNS server IP (in the lab, `10.1.1.50`).
- Loading the page is an **HTTP GET** request that flows host → switch → router →
  server and back, after DNS resolution completes.
- `show ip route` displays the router's routing table (its "map"): each network (shown
  in shorthand like `10.1.1.0/24`, meaning `10.1.1.0`–`.255`) and the interface it's
  reached through (e.g. directly connected to `Gig0/0` or `Gig0/1`). Routing = the
  router checking its map for the destination network and sending the packet out the
  matching interface.
- The lab's routing is very simplistic; real internet routers hold enormous tables.
  `show bgp ipv4 unicast` on a real internet router lists routes to vast numbers of
  networks. **BGP** is a routing protocol (covered later in the course).
- The real internet is many routers chained together (router → router → router), but the
  principle is the same: each router connects separate networks, and your home network
  has a different group of IP addresses from the rest of the world — which is why you
  need a router to reach the internet.
