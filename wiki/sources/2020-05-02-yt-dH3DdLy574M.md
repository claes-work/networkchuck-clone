# BLOCK EVERYTHING w/ PiHole on Docker, OpenDNS and IFTTT

## Summary

Part two of Chuck's network-blocking series (part one covered OpenDNS). This is a hands-on
self-hosting tutorial that runs Pi-hole inside a Docker container on a Linux host, points the
network's DNS at it for network-wide ad and content blocking, sets OpenDNS as the upstream DNS
resolver, and then adds voice-triggered blocking via a Python/Flask script exposed through IFTTT
and Amazon Alexa. The goal: block Netflix, Hulu, Disney+, ads, and "all kinds of distractions"
for you and your family — and toggle it on/off by voice ("Alexa, break everything" /
"Alexa, unblock my stuff"). Chuck runs his Linux host as a cloud VM (Ubuntu) but stresses the
same build works on a home VM, a physical machine, or a Raspberry Pi. He frames the combination
as a "perfect solution": Pi-hole for fast network-level ad blocking, OpenDNS for the content
filtering Pi-hole doesn't do natively. After ~a week running it in his own network he reports 26%
of all requests were blocked before ever using his ISP bandwidth.

## Key claims (dated — build steps + concepts)

All dated 2020-05-02 (video publication).

- This is part two; part one set up OpenDNS as a "very simple and elegant" way to protect a
  family's network. This video goes "nerdier and geekier" and combines OpenDNS with Pi-hole.
- Architecture: Pi-hole lives inside a Docker container, which runs on a Linux host. Chuck uses
  Ubuntu but says the beauty of Docker is it runs anywhere — any Linux flavor works.
- The Linux host can be a cloud VM, a home VM, a physical machine, or a Raspberry Pi. Chuck runs
  his in the cloud (spins up a new Ubuntu VM at a cloud provider, grabs its public IP).
- Chuck SSHes into the VM using Solar-PuTTY (his "favorite SSH program," free, affiliate link).
- Ubuntu conflict fix: later Ubuntu versions run their own DNS service (systemd-resolved) which
  conflicts with Pi-hole acting as the DNS server. Disable it with two commands:
  `sudo systemctl stop systemd-resolved` then `sudo systemctl disable systemd-resolved`.
- After disabling that service, DNS resolution breaks (ping google.com fails). Fix by editing
  `/etc/resolv.conf` (`sudo nano /etc/resolv.conf`) and changing the `nameserver` line — which was
  pointing at itself — to a working resolver (Chuck uses Google's for now; could be an OpenDNS
  server). Save with Ctrl+X, Yes.
- Install Docker: `sudo apt update` (updates the repo list, not an upgrade), then
  `sudo apt install docker.io`.
- Container creation uses `docker container run`, but with so many parameters Chuck uses a prepared
  bash script instead. The bulk of the script came from "the Pi-hole people"; he modified it and
  built a custom Docker image (uploaded to Docker Hub) that adds the Alexa/blocking functionality.
- Deploy the script: copy it from GitHub, create `pihole.sh` (`sudo nano pihole.sh`), paste, save.
- Running `./pihole.sh` fails with "permission denied" because the file isn't executable. Make it
  executable with `sudo chmod u+x pihole.sh` (chmod = change modification; `u+x` = add execute for
  the user), then run with `sudo ./pihole.sh`.
- The script pulls Chuck's (large) Docker image, starts the Pi-hole container, and prints a
  generated admin password.
- Verify the container: `sudo docker ps` shows the running container (healthy, ports mapped).
- Access Pi-hole via the host IP; the admin panel is at `<ip>/admin`. Log in with the generated
  password.
- Change the admin password by entering the container: `sudo docker exec -it pihole bash`, then
  run `pihole -a -p` and set a new password. (You know you're "in" when the shell prompt changes
  to the container ID.)
- Set the upstream DNS to OpenDNS in the web UI: Settings → DNS tab → select the OpenDNS upstream
  servers (or enter custom ones) → Save. Pi-hole forwards queries it doesn't block to this upstream.
- Point a client's DNS at the Pi-hole IP to route its queries through Pi-hole; verify with
  `nslookup google.com` / `nslookup facebook.com`. Queries then appear in the Pi-hole dashboard.
- Pi-hole has a whitelist and a blacklist. You can block manually (e.g. add Netflix as a regex),
  but Chuck automates it.
- Automation lives in the container's home directory under a `network` folder containing:
  `network.py` (a Python/Flask script — "the secret sauce"), `block-domains.sh`, and
  `unblock-domains.sh`.
- `block-domains.sh` runs `pihole --regex` with the domains to block (regex ensures everything
  related to Hulu/Netflix/Disney+ etc. is blocked); copy the format and substitute the middle term
  for any domain. `unblock-domains.sh` runs the near-identical command with `-b -d` (`-d` = delete,
  `-b` = block) to delete those entries from the blacklist.
- Run the Flask app: `sudo python3 network.py`. It serves on port 8080; hitting
  `<ip>:8080/block` blocks the domains and `<ip>:8080/unblock` unblocks them.
- Wire it to voice via IFTTT ("if this then that"): create an applet with Alexa as the trigger
  ("if this") on a specific phrase (e.g. "break everything"), and a Webhooks "make a web request"
  action ("that") pointing at the block URL.
- IFTTT can only reach a public URL. If the Linux server has a private (192.168.x.x) home IP,
  you must port-forward port 8080 on your router to the server's private IP and use your home's
  public IP (found via "what's my IP") in the webhook. Chuck's cloud VM already has a public IP.
- Create a matching applet for unblocking with whatever trigger phrase you want.
- Result: saying "Alexa, trigger break everything" blocks the sites; visiting Disney+/Hulu then
  fails, and the Pi-hole query log shows them blocked by regex.
- Verdict on Pi-hole's worth: after ~a week, 26% of all requests in his network were blocked (ads),
  before they even reached the internet or used ISP bandwidth. "It's wicked fast." Combined with
  OpenDNS for content filtering that Pi-hole doesn't do natively, it's a "perfect solution."
