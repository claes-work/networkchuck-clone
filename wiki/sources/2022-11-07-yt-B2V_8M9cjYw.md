# Block ALL ads with this…(and speed up your internet)

## Summary

A homelab/privacy tutorial in which Chuck sets up **AdGuard Home** — a self-hosted,
network-wide ad blocker — so that ads are blocked "at the source" for every device on
a home network rather than via a per-browser extension. He first explains the mechanism
using Wireshark: when you visit a site like cnet.com, your browser also fires DNS
requests to ad servers; a normal DNS server resolves those and the ads load. AdGuard
Home replaces your DNS server and answers ad-server lookups with nonsense (0.0.0.0) —
a technique called a **DNS sinkhole** — so the ads never load.

Chuck frames a second motive beyond ads: **privacy and control**. By default every
device uses the ISP's DNS server, letting the ISP see every site you visit; running
your own resolver takes that visibility back and lets you filter content (e.g. for
kids). He demos three deployment targets — a Raspberry Pi (headless install), a cloud
VPS (he mentions Linode/Akamai), and Docker (so it can run "anywhere") — then walks the
single install command, the setup wizard on port 3000, pointing the whole network at it
via the router's DHCP DNS setting, and DHCP reservation / static IP so the server's
address never changes. The advanced tour covers upstream DNS providers, DNS-over-HTTPS
(DoH), DNSSEC, upstream load-balancing, custom block lists, allowlists, DNS rewrites
(custom local hostnames), per-client policies, and blocking services (Facebook,
Instagram, Disney+). He positions AdGuard Home as more capable than Pi-hole,
specifically calling out DNS rewrites as a feature Pi-hole lacks.

## Key claims (dated — 2022-11-07)

Concept:
- Blocking ads network-wide beats a browser extension because it protects the whole
  network — every device, TV, Xbox, phone — not just one browser (paraphrase).
- Visiting a normal webpage triggers many DNS requests, including lookups to ad servers;
  a normal DNS server resolves those, which is how ads load (paraphrase).
- AdGuard Home works as your DNS server and responds to ad-server lookups with 0.0.0.0 /
  nonsense so the ad can't be reached and doesn't load — a technique called a **DNS
  sinkhole** (paraphrase).
- By default all home devices use the DNS server handed out by the ISP, so the ISP can
  see every website you visit; running your own DNS server restores privacy and gives
  you control over what the network can access (paraphrase).
- **DoH (DNS over HTTPS)** encrypts DNS queries from AdGuard to the upstream server, so
  neither the ISP nor anyone else can see the lookups (paraphrase).
- **DNSSEC** verifies every DNS response is legitimate, protecting against DNS poisoning
  / fake responses, a popular attack (paraphrase).
- AdGuard's default filter ships ~48,000 rules; more block lists can be added (paraphrase).
- AdGuard Home can also act as a DHCP server (in addition to DNS), though it should not
  replace the router (paraphrase).
- AdGuard Home supports DNS rewrites — custom local hostname → IP entries — which Chuck
  says Pi-hole does not do (paraphrase).

Setup:
- AdGuard is a company with several products; the specific one installed here is the
  free **AdGuard Home** (paraphrase).
- Deployment options shown: Raspberry Pi (headless), a cloud VPS, and Docker (runs
  anywhere) (paraphrase).
- Raspberry Pi headless install: flash **Raspberry Pi OS Lite (32-bit)** with the
  Raspberry Pi Imager, use the gear/settings icon to enable SSH, set a password, and
  configure Wi-Fi (or hard-wire); then boot and SSH in (paraphrase).
- Install steps: run `sudo apt update`, then paste the provided (long, automatic)
  install command; AdGuard Home installs and runs, exposing its GUI on **port 3000**
  (paraphrase).
- Setup wizard: mostly accept defaults, set a non-`admin` username/password (paraphrase).
- To actually use it, change the **DNS server your router hands out via DHCP** to the
  AdGuard box's IP (Chuck's example: 10.72.0.60) rather than changing DNS on each device
  individually (paraphrase).
- Ensure the AdGuard server's IP never changes via a **DHCP reservation** (easy) or a
  hard-coded static IP (harder) (paraphrase).
- Router DHCP lease is ~24 hours by default, so devices may take up to that long to pick
  up the new DNS server (paraphrase).
- Recommended upstream config: replace the default Quad9 entry with a Quad9 variant that
  includes a security block list + DNSSEC, plus Google and Cloudflare, and load-balance
  across all three; enable DNSSEC in DNS server configuration (paraphrase).
- Other GUI areas: general settings (parental controls, safe search, extend stats to 90
  days), per-client persistent policies, added block lists (e.g. Dan Pollock's list,
  game console ad block), allowlists, DNS rewrites, blocked services, and the query log
  (paraphrase).
- Claimed outcome: web browsing is ad-free, safer, and possibly a bit faster (paraphrase).
