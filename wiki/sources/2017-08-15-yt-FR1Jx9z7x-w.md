# Cisco Call Manager (CUCM) — Route Groups, Route Lists, Route Patterns!! (Part 2)

Part 2 of an early Cisco Unified Communications Manager (CUCM) call-routing tutorial.
Continues from Part 1 [[2017-08-11-yt-dQsaSdzfUAM]], where the route groups (named
"Luke" and "Leia" in a Star Wars-themed lab) were built. This part configures the
**route list**, then the **route pattern**, and tests outbound calling end to end.
The whole lab is framed around helping Captain Tarkin on the "Death Star" CUCM cluster
reach an external PSTN destination ("Tatooine").

## Summary

Chuck walks through the second half of the CUCM outbound call-routing chain, building
on the two route groups created in Part 1:

1. **Route list ("Empire route list").** He creates one route list under
   Call Routing → Route/Hunt → Route List, leaves "enable this route list" on, and
   checks the option to run it on all active Unified CM nodes (recommended in
   multi-node pub/sub environments for redundancy, even though this lab has only one
   node). He then adds both route groups in priority order: **Leia** as the primary
   path, **Luke** as failover.

2. **Per-route-group digit manipulation.** The key lesson of the video is that the
   called-party digit manipulation that would normally be configured on the route
   pattern can instead be configured **per route group inside the route list**. Because
   the two PSTNs have different formatting requirements, each route group gets its own
   transformation:
   - **Leia (primary):** discard digits "PreDot" (using the North American Numbering
     Plan / NANP discard instruction), then prefix `+1` to deliver the number in
     E.164 format, which that PSTN requires.
   - **Luke (failover):** a "weird / expensive" PSTN that requires the number prefixed
     with `818181` before it is sent.

3. **Ordering and overflow.** The up/down arrows in the route list set route-group
   order. As configured, Leia takes every call unless she is down, busy, or the maximum
   number of paths (e.g. concurrent SIP sessions on a paid SIP trunk) is exceeded — at
   which point calls overflow to Luke.

4. **Route pattern.** The route pattern is "a way out" of the CUCM cluster (internal
   calls between devices in the same calling search space / partitions route
   automatically; reaching the PSTN needs an explicit pattern). He builds one pattern
   `8.` (a leading `8` signals an external call; the dot separates the access code from
   the dialed digits), assigns it to the "Tatooine" partition to control who can dial
   it, and points it at the Empire route list. Because all the digit manipulation lives
   in the route list, **only one route pattern is needed** for both PSTNs — the payoff
   of the route-list approach.

5. **Route Plan Report** (Call Routing → Route Plan Report) is highlighted as the tool
   Chuck uses constantly to search and trace the whole configuration by partition —
   showing the pattern → route list → route groups chain and letting you click straight
   into each element.

6. **Testing in GNS3.** He labs two "cube" (CUBE / Cisco Unified Border Element) SIP
   routers in GNS3. A test call out the primary path shows the SIP INVITE arriving with
   the leading `8` stripped and `+1` added; after swapping priority to make Luke
   primary, the INVITE shows `818181` prepended — both from a single route pattern.

## Key claims (dated — the concept)

All dated 2017-08-15 (paraphrase unless quoted):

- A **route list** references one or more route groups in priority order; the first
  route group receives calls until it is unavailable (down/busy) or its path/session
  limit is exceeded, then calls overflow to the next group.
- Called-party digit manipulation can be applied **per route group within the route
  list**, rather than only on the route pattern — this lets a single route pattern
  serve multiple PSTNs with different formatting requirements.
- "Discard Digits: PreDot" (with the NANP instruction) removes the dialed access
  code/prefix up to the dot before the number is sent out; a prefix (e.g. `+1` or
  `818181`) can then be added.
- **E.164** is an international dialing format; delivering a North American number in
  E.164 here means prefixing `+1`. Meeting the PSTN's required format is the sender's
  responsibility.
- Enabling a route list on **all active Unified CM nodes** is the normal choice in
  multi-node (publisher + subscriber) deployments for redundancy.
- A **route pattern** is the "way out" of the CUCM cluster to an external destination;
  internal calls within a calling search space / partitions route automatically without
  a special pattern.
- The route pattern defines both which calls take the path and the dialing habit the
  user must dial; a leading `8` marks the call as external, and the pattern uses digit
  ranges (e.g. `2–9` in a given position).
- Assigning the route pattern to a **partition** controls which users/devices are
  allowed to dial it (via calling search spaces and partitions).
- The full outbound chain is: route pattern → route list → route group(s) → SIP
  trunk(s) → PSTN.
- The **Route Plan Report** (Call Routing → Route Plan Report) lets you search the
  entire configuration (e.g. by partition) and trace/click through pattern → route
  list → route groups.
- **GNS3** can be used to lab two CUBE SIP routers to test the routing end to end.
