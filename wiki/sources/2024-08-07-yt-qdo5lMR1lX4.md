# I forced EVERYONE to use Linux

## Summary

Chuck converts the studio's single, extremely powerful render/edit machine (previously
Windows 11, NVIDIA RTX 4090) into a **Linux terminal server** so his whole team can log
in and work on it simultaneously instead of fighting over one seat. The distro used is
**Ubuntu 20.04**. The terminal-services software is **ThinLinc** (by Cendio, the video's
sponsor), which he pitches as a free (up to 10 users) alternative to the expensive
Windows Server Terminal Services / RDS licensing he has paid for in the past.

The build has three parts, demoed by Chuck and his editor Alex: (1) install the ThinLinc
**master server** on the Ubuntu render machine and create Linux user accounts with
`adduser`; (2) install the ThinLinc **client** on user devices — anything works, including
a low-powered Raspberry Pi "thin client" — and connect over the network; (3) install
**VirtualGL** so remote sessions get server-side GPU (RTX 4090) hardware acceleration for
GPU-heavy work like Blender 3D modeling and video rendering. Chuck shows the ThinLinc web
admin console (system health, sessions, licenses, session termination, session
**shadowing**, and a desktop customizer to lock down what users can access). The payoff
demo: editor Mike runs Blender on a Raspberry Pi (via the server) to model "Koki," which
is then 3D printed. The core thesis is that you buy ONE powerful computer instead of many,
and cheap clients access it.

## Key claims (dated — stance + friction)

**Pro-Linux / pro-terminal-server stance (2024-08-07):**
- Chuck says he is "tired of" Windows taking over the office and wants to replace Windows
  and MacBooks with Linux across the team.
- A single sufficiently powerful PC is wasted under Windows 11 because only one user can
  work on it at a time; Linux terminal services lets ~5 people share it at once.
- Terminal services is not a new idea — Windows Server has had Windows Terminal Services
  for a long time — but Windows licensing is expensive (he recalls ~$1,000 for five users)
  and paying for it "hurt." Linux ThinLinc does the same thing free for up to 10 users.
- Adding users on Linux (`adduser`) is described as "so much easier than Windows."
- One powerful server + cheap thin clients (even a Raspberry Pi) beats buying many
  powerful computers — the client hardware doesn't matter because the heavy work runs on
  the server.
- ThinLinc gives per-user isolated sessions sharing the server's resources; the web admin
  console (system health, user lookup, load monitoring, session shadowing, desktop/app
  lockdown) is a favorite feature.

**Setup facts / requirements:**
- The ThinLinc server must be x86 (x86-64) — no ARM devices (so not a Raspberry Pi as the
  server); the Raspberry Pi is only ever the client.
- Sizing rule-of-thumb (attributed to Cendio): a full desktop with typical apps needs
  ~150–300 MHz CPU per active user and ~100–200 MB RAM per user as a bare minimum; scale
  up for heavy workloads. ThinLinc itself uses ~100 MB storage and each user profile
  ~100 KB.
- Client can be Windows, macOS, or Linux; dedicated thin terminals also exist (the kind
  seen at Taco Bell or doctor's offices).
- VirtualGL gives server-side hardware acceleration: without it the OpenGL renderer falls
  back to the software renderer "LLVMpipe" (using the CPU) at ~125 fps in glxspheres;
  running the same test through `vglrun` uses the NVIDIA RTX 4090 GPU and jumps to
  ~1,000 fps.

**Friction / honest caveats:**
- On some Linux setups the ThinLinc installer's GUI window appears; on Chuck's it fell
  back to text mode (he treats this as fine, "just as good").
- Near the end of install, ThinLinc commands are not on the sudo `secure_path`, which is
  "a little bit annoying" — fixed by editing `visudo` to add `/opt/thinlinc/bin` and
  `/opt/thinlinc/sbin`.
- Installing/configuring VirtualGL requires root (`sudo su -`), shutting down the display
  manager (e.g. `/etc/init.d/gdm3 stop`), finishing over SSH, and manually adding users to
  the `vglusers` group in `/etc/group` — display-manager steps differ per system, so
  "check the documentation."
- Running Blender on a Raspberry Pi client works but the Pi "has no ram" and Mike notes he
  "would usually want a lot more" for that kind of work — it runs, but it isn't ideal on
  the smallest client.
