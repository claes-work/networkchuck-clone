# 5 Steps to Secure Linux (protect from hackers)

## Summary

A hands-on Linux server hardening tutorial framed as the defensive counterpart to
Chuck's offensive/hacking content ("we're protecting ourselves"). Chuck argues that
anything can be hacked, but most real-world breaches happen on servers that were never
patched or secured — so basic hardening puts you "ahead of the game." He demonstrates
on a fresh Debian 10 cloud server (spun up on Linode, the sponsor), logging in over SSH
from Windows PowerShell (noting the steps are nearly identical on Mac/Linux). Credit is
given to "Awol," a moderator/Linux guru from his Discord, for helping build the video.

The core of the video walks through a sequence of hardening steps: enabling automatic
updates, replacing root login with a limited sudo user, switching from password auth to
an SSH public/private key pair, locking down the SSH daemon config (custom port, IPv4
only, no root login, no password auth), and finally standing up the UFW firewall
(allowing only the needed ports and blocking ICMP/ping to stay hidden). Chuck repeatedly
stresses testing each change from a second terminal before logging out so you don't lock
yourself out, and closes by reiterating that nothing is truly unhackable — this is best
practice, not a guarantee.

> Note on the "5 steps": the video is titled "5 Steps," but the on-screen narration
> explicitly numbers only Step 1 (updates), Step 2 (non-root sudo user), Step 3 (SSH
> keys), and Step 4 (lock down logins via sshd_config). The firewall (UFW) section is
> presented as the final hardening task ("one more thing") and is treated here as the
> fifth step. fail2ban is not actually covered in this transcript despite being a common
> item in this genre.

## Key claims (dated — the hardening steps)

All paraphrased from the 2021-03-18 video unless quoted.

- **Framing:** Anything can be hacked, but most big breaches happen in environments that
  were not properly secured or patched; doing basic hardening already puts you ahead.
- **Step 1 — Enable automatic updates (called the most important step).** Manually you
  can update with `apt update` (check repositories for available updates) then
  `apt upgrade` (install them). Because people forget to patch, install the
  `unattended-upgrades` utility (`apt install unattended-upgrades`) and configure it with
  `dpkg-reconfigure --priority=low unattended-upgrades`, choosing to auto-download and
  install stable updates. Unpatched servers are the most common way servers get hacked.
- **Step 2 — Stop using root; create a limited sudo user.** Never log in as root,
  especially over the internet. Create a normal user (`adduser networkchuck`), then add
  them to the sudo group (`usermod -aG sudo networkchuck`). The limited user can't run
  privileged commands directly (you'll get "command not found" because you can't even
  see the command exists), but prefacing a command with `sudo` borrows root's powers for
  that one command.
- **Step 3 — Use SSH key authentication instead of passwords ("passwords are for
  suckers").** Create a public/private key pair with `ssh-keygen -b 4096` (bigger key =
  more secure). Keep the private key hidden; upload the public key to the server. Prep the
  server with `mkdir ~/.ssh && chmod 700 ~/.ssh`, then copy the public key up — on Windows
  via `scp $env:USERPROFILE/.ssh/id_rsa.pub user@server:~/.ssh/authorized_keys`, on Linux
  via `ssh-copy-id user@server`, on Mac like the Windows way but with `~/.ssh/id_rsa.pub`.
  A key pair can't be brute-forced like a password, so it's much more secure.
- **Step 4 — Lock down logins by editing `/etc/ssh/sshd_config`** (with
  `sudo nano /etc/ssh/sshd_config`): change the default port 22 to something random/high
  (he uses 717) since everyone — including hackers — knows port 22; set AddressFamily to
  `inet` (IPv4 only, since most people don't need IPv6); set `PermitRootLogin no`; set
  `PasswordAuthentication no` so nobody can log in without a key. Then
  `sudo systemctl restart sshd`. Always test the new login from a second terminal before
  logging out, and connect on the custom port with `ssh -p 717 user@server`.
- **Step 5 — Set up a firewall with UFW (Uncomplicated Firewall).** First audit open
  ports with `sudo ss -tupln`; google anything unfamiliar and uninstall what you don't
  need. Install with `sudo apt install ufw`. Before enabling, allow your SSH port first
  (`sudo ufw allow 717`) so you don't lock yourself out, then `sudo ufw enable` (default
  denies everything else). Open additional services explicitly as needed (e.g.
  `sudo ufw allow 80/tcp` for a web server). To stay hidden, block ping/ICMP by adding a
  rule to `/etc/ufw/before.rules` (in the "ok icmp codes for input" section) and reload
  (`sudo ufw reload`, or reboot if the reload doesn't apply it) — blocking pings is a
  common best practice so a server doesn't announce it's up and ready for attacks.
- **Closing caveat:** Nothing is unhackable — even SSH keys aren't completely unhackable,
  just more secure than passwords. This is best practice, not foolproof.
