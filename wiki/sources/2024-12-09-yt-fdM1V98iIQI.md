# I built a phone system because no one answers me

## Summary

Chuck builds a full home + business VoIP phone system to solve a personal problem: no one in his family answers their cell phones. The stack is **3CX** (the video's sponsor), NOT Asterisk/FreePBX — he demonstrates both the free cloud-hosted 3CX SMB plan and an on-prem 3CX virtual machine running on his 45Drives Proxmox cluster. He wires up softphones (3CX app), supported SIP desk phones (Snom D735), and analog "grandpa" wall phones bridged in via a Grandstream HT801 ATA. Physical phones register to the 3CX cloud through a Raspberry Pi running 3CX's SBC (Session Border Controller) software. He adds outside connectivity via a VoIP.ms SIP trunk (a paid DID/phone number), builds an IVR / digital receptionist ("the Keith home" menu), and — the payoff — routes calls to his local AI voice assistant "Terry" via Home Assistant's VoIP integration so he can literally phone his AI and control his smart home. Throughout, he explicitly frames this as a callback to his former career as a Cisco phone system engineer.

## Key claims (dated — stack + roots)

- (2024-12-09) The phone system is built on **3CX**, not Asterisk/FreePBX; 3CX sponsored the video. The **3CX SMB plan is free forever for up to 10 users** and can be cloud-hosted by 3CX with no hardware required. (paraphrase)
- (2024-12-09) Cloud install is near-instant: signing up auto-provisions a hosted phone system server, credentials, and a number. Softphones register by scanning a QR code in the 3CX app. (paraphrase)
- (2024-12-09) Chuck also runs an **on-prem 3CX instance as a VM on his 45Drives cluster running Proxmox** for more features. (paraphrase)
- (2024-12-09) You can no longer install the full 3CX phone system on a Raspberry Pi; instead you install 3CX's **SBC (Session Border Controller)** software on a Pi to bridge local physical phones to the 3CX cloud. Officially supported on Pi 5; he ran it on a Pi 4. (paraphrase)
- (2024-12-09) Supported SIP desk phone used: **Snom D735**, powered over PoE, added by phone model + MAC address (3CX auto-provisions it). (paraphrase)
- (2024-12-09) Analog wall phones use POTS (plain old telephone service) with RJ11 and analog signals; modern phones use **SIP over RJ45/IP (VoIP)**. A **Grandstream HT801 ATA** (analog telephone adapter) translates analog ↔ SIP so old phones join the system. (paraphrase)
- (2024-12-09) Outside-world calling uses a **SIP trunk** to a phone-number provider; Chuck uses **VoIP.ms** (unsponsored, his preference), buys a DID (direct inward dial), and configures inbound routing + an outbound rule with a **prefix of 9** that is stripped before dialing — a habit from his phone-engineer days. (paraphrase)
- (2024-12-09) He builds an **IVR / digital receptionist** ("Pam," an Office reference) with a recorded menu routing callers to Chuck/Chloe/Addie by keypad. (paraphrase)
- (2024-12-09) The centerpiece: routing calls to his **local AI assistant "Terry" via Home Assistant's VoIP integration**. On-prem 3CX reaches it via a generic (unauthenticated, IP-based) SIP trunk to the Home Assistant server; cloud users instead point the Grandstream ATA directly at Home Assistant using off-hook auto-dial to the HA IP. Both require the **Opus codec** to be set as priority. (paraphrase)
- (2024-12-09) **Roots:** Chuck states he was formerly a phone system engineer — specifically a **Cisco phone system engineer** — which he says explains his love of phones and this build. The 3CX Call Flow Designer reminds him of tooling he used "back in the day." (paraphrase)
