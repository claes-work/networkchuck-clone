# CUCM Calling Search Spaces and Partitions: Explained with Star Wars

## Summary

An early NetworkChuck tutorial (first video in a planned Cisco Call Manager / dial
plan series) explaining Class of Control in Cisco Unified Communications Manager
(CUCM), implemented via **partitions** and **calling search spaces (CSS)**. Chuck
frames Class of Control as "permissions for the telephony world": a way to determine
who can call whom, and to restrict callers from reaching certain destinations (e.g.
international dialing, executives, 911).

The technical core: a **partition** is a container/grouping of dialable destinations
(directory numbers, route patterns, translation patterns, voicemail ports) — it
answers "who can call me." A **calling search space** is a group of partitions
assigned to devices or lines — it answers "who can I call." He warns against leaving
elements in the default `<None>` (null) partition, which grants everyone permission to
dial everything ("the Wild West, no control").

The entire lesson is taught through an extended **Star Wars analogy**: NetworkChuck is
"contracted by the Galactic Empire" to design a dial plan for stormtroopers (basic
users), captains (managers, e.g. Captain Tarkin), and C-suite (Darth Vader and the
Emperor). Geography maps to dial-plan scope: the Death Star = internal directory
numbers, Tatooine = local outbound (dial 9), and Serpent Doll (a planet in the outer
rim) = intergalactic/international dialing. He shows two design approaches, plus
line-vs-device CSS precedence and translation-pattern call blocking (the "Force Choke"
CSS). Notable as an early, clear example of Chuck's signature **analogy-driven
teaching style**.

## Key claims (dated — the concept + analogy)

All dated 2017-01-24 (video publication). All paraphrase unless quoted below.

- Class of Control is basically permissions for the telephony world — a way to control
  who can call whom; if you don't want callers reaching certain destinations, you
  restrict them with a restrictive CSS in Call Manager. It is implemented via
  partitions and calling search spaces.
- A voice/network engineer needs to know the dial plan intimately: how calls flow, and
  how to restrict users from dialing internationally, dialing executives, or dialing
  911.
- A **partition** is a container/grouping of dialable destinations — directory numbers,
  route patterns, translation patterns, voicemail ports — grouped together in CUCM.
  Partitions answer "who can call me."
- A **calling search space (CSS)** is a group of partitions; you assign partitions to a
  CSS, and the CSS is assigned to any device (or line) that can dial other things. CSS
  answers "who can I call."
- The default null partition shows up in Call Manager as `<None>`; every phone and
  dialable element sits there by default after a base install, giving zero permissions
  and zero security. You should get everything out of the null partition ("it's the
  Wild West, no control").
- Analogy mapping: stormtroopers = basic users; captains (Captain Tarkin) = managers;
  Darth Vader and the Emperor = C-suite. Death Star = internal directory numbers;
  Tatooine = local outbound (dial 9 + pattern); Serpent Doll (outer rim) =
  intergalactic / international dialing (route pattern `9.011` + international
  destination).
- Design requirements in the analogy: stormtroopers may dial only within the Death Star
  (internal) and reach managers, but not Tatooine, not Serpent Doll, and not the
  C-suite; captains may dial everyone in the Death Star plus Tatooine, but not Serpent
  Doll; Darth Vader and the Emperor get full access to everything.
- Design approach #1 — per-role CSS: create a stormtrooper CSS (Death Star partition
  only), captain CSS (Death Star + Tatooine), and Dark Side CSS (Death Star + Tatooine
  + Serpent Doll), each assigned to the matching phones.
- Blocking within a partition: because a device CSS granting the Death Star partition
  would let a stormtrooper dial the Dark Lords, you create a "Dark Lord block"
  partition containing **translation patterns** for Vader's and the Emperor's directory
  numbers set to **block** (rather than route) the calls, put that in a "Force Choke"
  CSS, and assign that CSS to the stormtrooper's **line**.
- **Line vs. device CSS precedence**: when both a line CSS and a phone/device CSS are
  present, they concatenate ("gel"), and the **line CSS has priority** — its rules
  override the device CSS. So a blocked pattern on the line CSS is blocked even though
  the device CSS would otherwise allow it; patterns that don't match the line CSS fall
  through to the device CSS.
- Design approach #2 — one broad phone CSS ("Empire CSS") granting access to all
  dialing, assigned to every phone, then restrict per role using line CSSs with block
  partitions (e.g. Tatooine block, Serpent Doll block). The line CSS blocks matching
  patterns first, then anything not already blocked falls through to the phone CSS.
- Important caveat: two directory numbers in the **same partition cannot necessarily
  dial each other** — reachability is determined by the caller's CSS, not by shared
  partition membership. A "contractor stormtrooper" whose CSS holds only the Serpent
  Doll partition cannot dial a fellow stormtrooper in the Death Star partition, even
  though that fellow (with a Death Star CSS) can dial him.
- Configuration path in CUCM: **Call Routing > Class of Control** — create partitions
  first (can be added in bulk, separated by lines), then create calling search spaces
  and move the desired partitions into the "Selected Partitions" box; the "Copy"
  button lets you retain common partitions across new CSSs.
- Framing of the role: to be the best voice/network engineer you often have to be both
  — voice engineers are usually network engineers and vice versa.
