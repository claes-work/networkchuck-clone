# bad USBs are SCARY!! (build one with a Raspberry Pi Pico for $8)

## Summary

Chuck demonstrates and explains the "BadUSB" attack class: a device that looks like an
ordinary USB flash drive but is actually a keystroke-injection tool. He shows it live
by plugging one into his own laptop — with hands off — and watching it disable Windows
Defender, open a reverse shell, and run a locked-volume rickroll, all automatically.

The core teaching point is *how* the attack works: the BadUSB does not present itself
as mass storage but as an HID (Human Interface Device) — a keyboard/mouse — which the
computer trusts by default. Because it "types" around 1,000 words per minute, it
finishes its payload before a human notices. Chuck frames this as both a hacker tool
(devices left in parking lots for curious people to plug in) and, crucially, a
defensive awareness lesson.

He then walks two builds: the commercial USB Rubber Ducky from Hak5 (~$45), and a
cheap DIY alternative — a ~$5–8 Raspberry Pi Pico flashed with CircuitPython and the
Adafruit HID library, running the community "pico ducky" project (by dabisu) so it
executes DuckyScript payloads. He covers writing/encoding payloads (Ducky Encoder for
the Rubber Ducky; plain `payload.dd` for the Pico) and a flash_nuke reset workaround.

The final third is defense: don't plug in unknown USB drives, lock your computer when
you walk away (Windows+L), require a password for administrator (UAC) prompts, apply
least privilege (no admin rights for regular users), and optionally physically block or
disable USB ports via policy. The explicit ethics framing: learn the attack so you can
defend yourself, your family, and your company.

## Key claims (dated — concept + defense)

Concept / how it works (2021-08-20, paraphrase):
- A "BadUSB" looks like an ordinary flash drive but is built to attack the computer it's
  plugged into; it is a keystroke-injection device, not storage.
- The trick is that it registers as an HID (Human Interface Device) — a keyboard/mouse —
  and computers trust human-interface input by default, so it is inherently trusted.
- It types far faster than a human (Chuck estimates ~1,000 words per minute vs. his own
  ~50–75 wpm), so it completes its payload before the victim can react.
- DuckyScript (designed by Darren Kitchen of Hak5) is just simple commands that simulate
  keystrokes: DELAY, key presses like CTRL/ESC, STRING to type text, ENTER, TAB, etc.
- Attacks work across Windows, macOS, and Linux because any HID keyboard/mouse input can
  be mimicked — "if you can do it on a keyboard you can do it with this."
- A common delivery vector is social: hackers leave BadUSBs in parking lots or on desks
  hoping a curious person plugs one in.

Cheap-hardware build (2021-08-20, paraphrase):
- A ~$5 Raspberry Pi Pico can be turned into a BadUSB, versus ~$45 for the Hak5 USB
  Rubber Ducky.
- Setup: flash CircuitPython (drag the `.uf2` onto the Pico), add the Adafruit HID
  library folder into `lib`, then load dabisu's "pico ducky" script and rename it to
  `code.py` so it auto-runs on plug-in.
- The Rubber Ducky requires encoding payloads to `inject.bin` (via a simple GUI Ducky
  encoder); the Pico runs a plain-text `payload.dd` with no encoding.
- Downsides of the Pico: it needs a cable (less covert than a flash-drive form factor),
  shows up as both mass storage and HID, and re-editing a payload requires a
  boot-select + `flash_nuke` factory reset because it runs the payload on every plug-in.

Defense (2021-08-20, paraphrase):
- Don't plug unknown/found USB devices into your computer — it can be compromised in
  seconds (malware, data exfiltration, or "just" a rickroll).
- Always lock your computer when you step away (Windows+L); a locked machine that needs a
  password defeats the attack because the device doesn't know your password.
- Because user education fails to human error, add technical controls: require a password
  for administrator/UAC prompts (registry `ConsentPromptBehaviorAdmin` = 1, best rolled
  out via Group Policy).
- Apply least privilege — don't give regular users local admin; many payloads need
  admin, so without it the device is rendered useless.
- Optional physical/policy controls: lockable USB-port blockers or disabling USB ports
  entirely (inconvenient, since USB is used for everyday devices).
