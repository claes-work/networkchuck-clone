# how to HACK a password // Windows Edition

## Summary

A hands-on ethical-hacking tutorial in which Chuck demonstrates, against a coworker
"Michael" (with explicit permission — Chuck frames himself as Michael's boss who "makes
the rules"), the full chain of extracting and cracking a Windows login password. The
core lesson: Windows never stores the plaintext password. It stores a **hash** (Chuck
names Microsoft's algorithm as MD4, underlying NTLM), and if an attacker can grab that
hash they can either crack it into the plaintext password or, more alarmingly, use the
hash directly to authenticate.

Chuck walks the workflow conceptually: with brief physical access to an unlocked admin
machine, he pulls the SAM and SYSTEM registry hives (`reg save` of `HKLM\SAM` and
`HKLM\SYSTEM`), exfiltrates them, then on a Kali Linux box uses Impacket's
`secretsdump` to parse out the NTLM hashes. He cracks the hash with a dictionary
attack: generating a candidate wordlist with the `cupp` tool (interactive mode, fed
personal details like name, birthday, partner, pets, hobbies) and running it through
**hashcat** (mode 1000 = NTLM). Once cracked he logs in remotely via `evil-winrm` and
`xfreerdp`. He then shows **pass-the-hash** — authenticating with the raw hash instead
of the password at all (capital `-H` in evil-winrm; `/pth` in xfreerdp).

The back half is the defensive turn. Chuck maps the whole attack to the **MITRE ATT&CK**
framework (OS Credential Dumping via Security Account Manager), noting the framework
even names the tools and offers mitigations: disabling/restricting NTLM, password
policies, user training, and detection of suspicious registry activity. He also gives
full disclosure that several security features had to be manually weakened on Michael's
PC for the remote steps to work (disabled Windows firewall, enabled Remote Desktop,
added Michael to allowed RDP users, disabled a restricted-admin registry setting) —
grabbing the hash needed nothing, but Windows' default security posture blocked the
remote access. Framed throughout as lab-only and permission-only. Sponsored by Dashlane
(password manager) with the practical takeaway to use strong, unique passwords.

## Key claims (paraphrase; dated 2023-06-30)

**Concept:**
- Windows does not store passwords in plain text; it stores a hash produced by a hashing
  algorithm (Chuck names Microsoft's as MD4, underlying NTLM). At login the entered
  password is re-hashed and compared to the stored hash.
- The password hashes live in the Windows registry across two hives — SAM and SYSTEM —
  under `HKLM`; both are needed to extract usable hashes. Administrator rights (a UAC
  prompt) are required to read them.
- Impacket's `secretsdump` parses the saved SAM + SYSTEM files locally to output the
  NTLM hashes for each account.
- Cracking a hash is a guessing game: try many candidate passwords, hash each, and
  compare. A **dictionary attack** feeds a large wordlist (thousands of candidates);
  it is what most attackers use, and breached data typically surfaces as email +
  password hash pairs.
- `cupp` generates a targeted wordlist from personal details about the victim (name,
  birthday, partner, pets, company, hobbies), producing tens of thousands of plausible
  passwords instantly.
- `hashcat` with mode 1000 (NTLM) runs the wordlist against the hash and reports the
  cracked plaintext.
- **Pass-the-hash**: an attacker can authenticate with the raw NTLM hash directly,
  without ever cracking the password — demonstrated over both WinRM and RDP.

**Defense / mitigation:**
- The attack is well-documented in the MITRE ATT&CK framework (OS Credential Dumping →
  Security Account Manager), which lists both the technique and mitigations.
- Recommended mitigations: disable or restrict NTLM, enforce password policies, train
  users, and enable detection of suspicious registry-key access.
- Strong, unique passwords per service are the primary defense; reused/weak passwords
  are "probably the main reason people get hacked." A password manager (video sponsor:
  Dashlane) helps generate and enforce them.
- Windows' default security posture is "pretty good": the remote-access steps only
  worked after manually disabling the firewall, enabling RDP, adding the user to allowed
  RDP users, and disabling a restricted-admin registry setting. Grabbing the hash itself
  required no such changes.
