# if if if if if....elif

Episode 5 of NetworkChuck's free "Python right now" course, continuing the flow-control
thread from the previous episode. Chuck teaches nested `if` statements and the `elif`
keyword using the running "coffee shop robot barista" narrative (screening evil Bens,
pricing menu items). Learners follow along in a free browser-based Python lab.

## Summary

The episode builds on episode 4's flow control (a single `if/else` that kept "evil Ben"
out of the coffee shop). Chuck's new premise: not all Bens are evil, so the script needs
to interrogate a visitor before deciding. He introduces **nested ifs** — an inner `if`
placed (and indented) under an outer `if`, so the inner check only runs when the outer
condition is true. The flow: outer `if name == "Ben"` asks a follow-up question via
`input()` ("are you evil?"), stores the answer in `evil_status`, then an inner
`if evil_status == "yes"` kicks him out. He stresses that indentation alone determines
what is nested under what, and demonstrates the bug of leaving code at the wrong
indentation level (a print running for *any* Ben regardless of evil status). He then nests
an `else` under the inner `if` to greet non-evil Bens, and shows that the outer `else`
(aligned with the outer `if`) handles everyone who isn't Ben (e.g., Brittany). He notes
you can nest ifs arbitrarily deep but shouldn't — it gets messy.

The second half introduces **`elif`** ("else if"). Using the menu-pricing problem
(different drinks should cost different amounts), he first shows the "messy way": a stack
of independent, non-nested `if` statements each setting a `price` variable. He explains
why nesting would be wrong here (an order can't be both a frappuccino *and* a black
coffee) and why a trailing `else` inside that stack breaks the price (a later branch
overwrites the correct earlier value). The clean solution is an `if / elif / elif / ... /
else` chain: check frappuccino → `$13`, `elif` black coffee → `$3`, `elif` espresso →
`$5`, and a final `else` for unknown orders. Only the first matching branch runs. He hits
a live "price is not defined" `NameError` when a green-tea order matches no branch, and
fixes it by setting `price = 0` in the final `else`. He closes with a challenge lab
(latte + optional whipped cream → `$11` if yes, else `$9`) combining nested ifs and elif.

## Key claims (paraphrase; dated — concepts)

- (2022-02-17) A **nested if** is an `if` placed and indented underneath another `if`; the
  inner block runs only when both the outer and inner conditions are true. [para]
- (2022-02-17) **Indentation/spacing is what defines nesting** in Python — code is nested
  under a statement because it is indented beneath it; getting the spacing wrong changes
  which condition governs the code. [para]
- (2022-02-17) If a print statement is indented only under the outer `if name == "Ben"`
  (not under the inner evil-status `if`), it runs for every Ben regardless of evil status;
  to gate it correctly you must nest it under the inner `if`. [para]
- (2022-02-17) An `else` is tied to a specific `if` by its indentation level — an `else`
  aligned with the outer `if` runs when the outer condition is false; a nested `else` runs
  when the inner condition is false. [para]
- (2022-02-17) You can nest ifs as deeply as you want, but you shouldn't — it becomes messy
  and confusing. Nested ifs are most useful when you need to confirm that multiple things
  are true at the same time. [para]
- (2022-02-17) The `input()` function prompts the user and returns their typed answer,
  which can be stored in a variable (e.g., `evil_status = input("are you evil?\n")`) and
  then tested. [para]
- (2022-02-17) String comparison uses the double-equals operator `==` (e.g.,
  `evil_status == "yes"`), distinct from the single `=` used for assignment. [para]
- (2022-02-17) `elif` ("else if") lets you test an alternative condition when the previous
  `if`/`elif` was false; branches are on equal indentation footing (not nested), and only
  the first matching branch executes. [para]
- (2022-02-17) A chain of independent `if` statements can produce wrong results because a
  later branch's `else` can overwrite a value already set by an earlier true branch;
  `if/elif/else` avoids this because only one branch runs. [para]
- (2022-02-17) If a variable is only assigned inside conditional branches and no branch
  matches, referencing it later raises a `NameError: name '...' is not defined`; providing
  a final `else` that assigns a default (e.g., `price = 0`) prevents this. [para]
- (2022-02-17) `\n` inside a string produces a newline for spacing in prompts/output. [para]

  episode ships with a free browser-based Python lab; the best way to learn is to actually
