# EXPOSE your home network to the INTERNET!! (it's safe)

## Summary

A step-by-step tutorial on securely exposing self-hosted home-network services to
the public internet using **Cloudflare Tunnel** — a free service — instead of the
traditional approach of port-forwarding ports 80/443 and running a reverse proxy
(Nginx, Traefik) that punches holes through your firewall. The core idea: install a
lightweight tunnel connector on any computer/server inside the home network, which
dials **outbound** to Cloudflare; Cloudflare then routes chosen subdomains to internal
services through that tunnel. Because the connection is initiated from inside, no
inbound firewall ports are opened, and it works even behind CGNAT or a restrictive ISP.
Each exposed service automatically gets its own subdomain and a valid SSL certificate,
even if the internal service only has a self-signed or HTTP cert.

Chuck walks through the full setup: buying a domain (~$12/yr via Google Domains),
creating a Cloudflare account, pointing the domain's name servers to Cloudflare,
enabling Cloudflare Zero Trust (free plan), creating a tunnel, and running the connector
(his preference: the Docker install, adding `-d` to daemonize). He then routes two
services — a Proxmox server (`https://` to internal IP `10.7.7.3:8006`) and a Synology
NAS (`https://…:5001`) — using "No TLS Verify" for services with self-signed certs. He
validates access both from inside and from an external Kasm workspace hosted on Linode.
The video is sponsored by Red Hat, which Chuck uses as the tunnel host (Red Hat
Enterprise Linux via the free Developer subscription, licensing up to 16 servers).

## Key claims (dated — concept + security)

- [2022-12-14] Cloudflare Tunnel lets you securely expose anything on a home network
  (website, Proxmox, NAS, etc.) to the internet without port-forwarding or opening
  firewall ports 80/443 — paraphrase.
- [2022-12-14] The mechanism is a **tunnel**: a connector installed inside the home
  network dials **outbound** to Cloudflare, and Cloudflare routes subdomains back through
  it — so no inbound holes are punched through the firewall — paraphrase.
- [2022-12-14] Because the tunnel is outbound-initiated, it works regardless of network
  type, including behind CGNAT or a bad ISP — paraphrase.
- [2022-12-14] Every exposed service automatically gets its own subdomain and a valid
  SSL certificate, even if the internal service has only an HTTP or self-signed cert —
  paraphrase.
- [2022-12-14] Cloudflare Tunnel acts as a reverse proxy, but unlike Nginx/Traefik it
  does not insecurely expose the home network directly to the internet — paraphrase
  (security angle).
- [2022-12-14] Requirements are minimal: a domain name (~$12/yr) and any computer/server
  inside the home network running practically any OS — paraphrase.
- [2022-12-14] Setup uses Cloudflare Zero Trust on the free plan; the connector can be
  installed many ways, and Chuck prefers the Docker snippet, adding `-d` to run it as a
  background daemon — paraphrase.
- [2022-12-14] For services with self-signed certificates (e.g. Proxmox), enable "No TLS
  Verify" under the tunnel's additional application / TLS settings — paraphrase.
- [2022-12-14] With this setup you avoid managing Let's Encrypt certificates yourself,
  and things like the Proxmox console (which is painful to proxy through Nginx/Traefik)
  work out of the box — paraphrase.
- [2022-12-14] To use Cloudflare as name servers, change your registrar's custom name
  servers to the two Cloudflare name servers assigned to your domain — paraphrase.
