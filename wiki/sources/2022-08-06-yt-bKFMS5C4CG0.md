# Docker networking is CRAZY!! (you NEED to learn it)

## Summary

A hands-on Docker networking tutorial where Chuck walks through all **seven** network
types Docker can deploy, framing the whole video as a "rabbit hole" for his networking
background. Working in a free Ubuntu desktop VM (VirtualBox, type-2 hypervisor) with
`docker.io` installed, he builds a lab of BusyBox and NGINX containers (named after
Thor characters — Thor, Mjölnir, Storm Breaker, Loki, Odin, Jane Foster) to
demonstrate each network type live in the terminal.

The through-line is his core identity: Docker networking is really just networking
concepts (bridges, switches, VLANs, routing, layer 2 vs. layer 3) applied to
containers, so a networking person already has the mental model. He explicitly ties
the harder modes (802.1Q trunking, "router on a stick," layer-3 routed networks) to
classic network engineering, saying they're a "love letter to networking nerds."

Covers, in order: **default bridge**, **user-defined bridge**, **host**, **macvlan**
(bridge mode + 802.1Q mode), **ipvlan** (L2 + L3), **overlay** (mentioned only), and
**none**. The video credits Christian from "That Digital Life" for the macvlan/ipvlan
usage tips, and is sponsored by Bitdefender.

## Key claims (dated — the networking concepts)

All from the 2022-08-06 video; all paraphrase unless quoted below.

- **There are seven Docker network types.** Docker ships with three networks by default
  (bridge, host, none) — visible via `docker network ls`. In `docker network ls`,
  the "driver" column effectively means the network type.
- **Default bridge (`docker0`).** Installing Docker creates a virtual bridge interface
  called `docker0` with its own IP/subnet (the demo shows `172.17.x` range).
  Containers launched without a `--network` flag land on the default bridge. Docker
  auto-creates one virtual ethernet (veth) interface per container and links them all
  to `docker0`, which behaves like a switch; `bridge link` shows the veths tied to
  `docker0`.
- **Bridge does DHCP, DNS, and NAT.** `docker0` hands out IP addresses (acts as a DHCP
  server), copies the host's `/etc/resolv.conf` into each container so they share DNS,
  and gives containers a default route/gateway of `docker0`. Containers reach the
  internet via NAT masquerade (`nat masquerade`). Containers on the same bridge can
  ping each other by IP.
- **Default bridge requires manual port exposure.** Services inside a container (e.g.
  an NGINX web server on port 80) are NOT reachable from the host/LAN by default. You
  must publish the port with `-p 80:80` (container port : host port) and redeploy. This
  isolation is a downside but also a security benefit. Inspect a network with
  `docker network inspect bridge`.
- **User-defined bridge is what Docker recommends.** Docker discourages using the
  default bridge and wants you to create your own with `docker network create <name>`.
  A user-defined bridge is functionally the same as the default bridge but adds two
  benefits: (1) **isolation** — separate user-defined bridges cannot talk to each other
  or to the default bridge; (2) **automatic container-to-container DNS** — containers on
  a user-defined bridge can ping each other by container name (useful because container
  IPs change on redeploy). The default bridge does NOT give name-based DNS.
- **Host network.** Deploying with `--network host` gives the container no separate
  network of its own — it shares the host's IP address and ports directly, running
  essentially like a normal application installed on the host. No port exposure is
  needed. Downside: no isolation. Chuck's example use case (via Christian) is a
  WireGuard VPN container you want directly on the host network.
- **Macvlan.** Connects containers directly to the physical LAN; each container gets its
  own MAC address and its own real IP on the home network, behaving like a virtual
  machine on the LAN. Created with `docker network create -d macvlan --subnet <lan> \
  --gateway <router> -o parent=<host-nic> <name>`. You must tie the macvlan to a
  physical NIC (`parent=` the host interface, e.g. `enp0s3`). Requires the VM be on a
  bridged adapter (not NAT) so it sits on the real LAN.
- **Macvlan downside 1 — promiscuous mode.** Because each container gets its own MAC,
  multiple MACs appear on one switch port; many switch ports have port security limiting
  MACs per port, which breaks it. Fix: enable **promiscuous mode** on the host NIC
  (`ip link set <nic> promisc on`) — and in the demo also in VirtualBox settings
  (network > advanced > promiscuous mode: allow all) plus a host reboot. On real
  hardware you may have to enable it on each network device.
- **Macvlan downside 2 — no DHCP.** Macvlan does not get an IP from your router's DHCP;
  worse, Docker runs its own DHCP and may assign conflicting low addresses (.2, .3, .4).
  Best practice: manually assign each container's IP with `--ip`, keeping it outside the
  router's DHCP range, or restrict Docker's assignable range to a single unused IP
  (Christian's method).
- **Macvlan has two modes.** Bridge mode (the default, acts like a bridge onto the LAN)
  and **802.1Q mode**, where specifying a sub-interface like `enp0s3.20` / `enp0s3.30`
  makes Docker auto-create sub-interfaces and send those VLANs over the link like a
  trunk — "router on a stick" with containers. Requires trunking configured on the
  upstream switch.
- **Ipvlan L2.** Nearly identical to macvlan but solves the promiscuous-mode problem:
  containers share the **host's MAC address** while still getting their own real IPs on
  the LAN, so the switch sees one MAC with many IPs. Created with
  `docker network create -d ipvlan ...`; L2 is the default mode. Still has the manual-IP
  issue. Verified via the host ARP table showing the container IP mapped to the host's
  MAC.
- **Ipvlan L3.** Turns the host into a **router**: containers connect to the host as if
  it were a router, purely layer-3 (no switching, no ARP, no broadcast traffic). You can
  define multiple brand-new subnets that don't exist on the LAN. The gateway is NOT
  specified — it's the parent interface. Options: `-o parent=<nic> -o ipvlan_mode=l3`,
  and multiple `--subnet` flags for multiple networks. Containers can't reach the
  internet or be reached by the LAN until you add a **static route** on your router
  pointing those subnets' next-hop to the host's IP. Networks sharing the same parent
  interface can talk to each other; for isolation, use different physical interfaces.
  No exposed ports — the concept doesn't apply at L3. Praised as network best practice
  (push layer-3 to top-of-rack, avoid layer-2 bridging loops / BPDUs / spanning tree).
- **Overlay (mentioned, not demoed).** For multiple hosts each running containers
  (e.g. Docker Swarm, Docker's version of Kubernetes). Overlay abstracts away the
  cross-host networking complexity and lets you set rules for how containers on
  different hosts talk. Not covered in-video.
- **None network.** Driver is `null`; already exists, nothing to create. A container on
  `--network none` has no networking at all — only a loopback interface. Presented as
  the "most secure" network.
- **Career note.** Chuck advises putting "Docker networks" as a skill on your resume and
  doing this lab.
