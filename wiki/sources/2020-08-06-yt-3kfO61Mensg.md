# REAL LIFE example!! (TCP/IP and OSI layers) // FREE CCNA // EP 4

## Summary

Episode 4 of Chuck Keith's FREE CCNA course (sponsored by Boson Software).
Chuck uses Cisco Packet Tracer in simulation mode to follow a single packet
through a network in "bullet time," showing how it interacts with each layer of
the networking models. The running scenario: "Johnny" wants to buy NetworkChuck
coffee, so his laptop sends web traffic (HTTPS) across the network to the
`networkchuck.coffee` server — through a switch, a router, and another switch.

Chuck notes that the real transport in the demo uses the four-layer TCP/IP
model, but for teaching he stacks on the extra OSI layers (session and
presentation) so the full seven-layer model is visible. He filters Packet
Tracer to show only HTTP/HTTPS traffic, so ARP and DNS steps from the prior
episode are hidden. He encourages viewers to download the linked Packet Tracer
file and follow along on their own computer.

The core lesson is **encapsulation**: as data moves down the stack, each layer
wraps it in an additional header (like putting a letter inside successive
envelopes), and the message gets a new name at each stage — segment (layer 4),
packet (layer 3), frame (layer 2). At each hop, devices see only the layers they
operate at (a switch sees only layers 1–2), and the destination server
**de-encapsulates** the frame layer by layer until it reaches the application
data. Chuck closes with two exam-style quiz questions (one old CCNA, one old
CCNP ROUTE) walked through with the process-of-elimination method, and previews
that upcoming episodes will dive deeper into the upper layers.

## Key claims (dated — the concepts)

_All 2020-08-06, paraphrased from Chuck._

- In reality the demo uses the four-layer TCP/IP model, but Chuck teaches with
  the seven-layer OSI model by adding the session and presentation layers for
  reference; the networking world "always" refers to the seven layers.
- The device's goal is to push the data all the way down through the layers to
  the physical layer so it can travel across the Ethernet cable to the other
  server.
- **Layer 7 (application):** the layer of the application in use. Johnny's web
  browser is the application; the protocol is HTTP — specifically HTTPS because
  it is secure and encrypted. HTTP is an example of an application-layer
  protocol.
- **Layer 4 (transport):** answers "how do we transport the data." Two main
  options: TCP (more reliable, has mechanisms to resend if data doesn't arrive)
  and UDP (less reliable but typically faster). This layer also carries ports —
  destination port 443 is reserved for HTTPS; port 80 is HTTP.
- **Encapsulation** is the process of taking data and attaching a layer header
  to it (e.g., adding the layer 4 header to the application data). It happens at
  every layer down the stack — like putting a letter inside another envelope
  repeatedly.
- Naming by layer: application data wrapped with a layer 4 header is a
  **segment**; wrapped with a layer 3 header it is a **packet**; wrapped with a
  layer 2 header (and trailer) it is a **frame**.
- **Layer 3 (network):** deals with IP addresses and routers. The layer 3 header
  contains source and destination IP addresses and gives the router directions.
  In the demo, source IP 10.1.1.3 (Johnny) → destination IP 23.227.38.65
  (`networkchuck.coffee`).
- **Layer 2 (data link):** deals with MAC addresses and switches; the layer 2
  header gives the switch directions to the next hop (source = Johnny's MAC,
  destination = the router's MAC). Layer 2 has both a header and a trailer.
- A switch reads only layers 1–2: it opens the frame's outer envelope, looks up
  the destination MAC in its CAM table (MAC address table), and forwards out the
  matching port.
- A router **de-encapsulates** to layer 3, reads the destination IP, consults its
  routing table, then re-encapsulates into a new layer 2 frame — rewriting the
  source and destination MAC addresses (now source = router, destination =
  `networkchuck.coffee`) — before sending it on. The router learns the next
  device's MAC address using ARP.
- The destination server (built to the TCP/IP model) de-encapsulates the frame
  layer by layer: checks the layer 2 MAC is its own, then the layer 3 IP, then
  layer 4 (TCP, port 443), then hands the data up to the application layer
  (HTTPS / browser info). It then reads the request and the whole process
  repeats in reverse to send the web page back.
- In the OSI model, application-layer (layer 7) functions include managing
  communications between applications and directing data to the correct
  programs; managing logical/physical addressing is layer 2, and ensuring
  error-free delivery is the transport layer (TCP). "Ensuring error-free data
  delivery" quiz option maps to the presentation layer (layer 6) under OSI, but
  in TCP/IP layers 5, 6, and 7 are clumped into the application layer.
- TCP relies on a **three-way handshake** to provide connection-oriented,
  reliable data transfer; UDP does not (it is fast but not reliable). ARP, DNS,
  and RARP are not transport protocols — ARP maps an IP to a MAC, DNS maps a
  friendly name to an IP, and RARP (reverse ARP) finds an IP from a MAC.
