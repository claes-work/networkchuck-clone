# you need to learn Ansible RIGHT NOW!! (Linux Automation)

## Summary

A landmark entry in Chuck Keith's signature "you need to learn X RIGHT NOW" tutorial
series, this one on **Ansible** for IT automation. The video follows the format's usual
arc: a hard-hook title, a "why should you care" pitch, a coffee cue, then a hands-on lab.
Chuck frames Ansible as one of the most popular IT automation tools — for automating
systems, cloud, and networking — owned by Red Hat, easy to use, and not requiring
programming knowledge.

He motivates the tool from a Linux sysadmin's pain: managing one server is easy (edit
`/etc/resolv.conf`), but managing many servers means logging into each box individually to
make the same monotonous change. Ansible solves this with a **control station** (master
node) that pushes changes out to all managed servers at once. The same pattern applies to
network engineers managing many Cisco routers/switches.

The lab (sponsored by Linode) spins up three CentOS 7 servers: one control node and two
managed nodes. Chuck installs Ansible (yum update → EPEL release → install ansible),
edits the **inventory/host file** (`/etc/ansible/hosts`) to define a `[linux]` group with
connection variables, tweaks `ansible.cfg` (disable host key checking for the lab), then
runs ad-hoc commands (`ping` module, `-a` ad-hoc shell commands, reboot) and finally a
**playbook** (a YAML file) that installs Nano via the yum module. He demonstrates
**idempotence** by running the playbook twice (second run = zero changes) and reverses it
by changing state from `latest` to `absent`. Closes by pointing to Ansible module docs,
the CBT Nuggets Ansible course, and teases a Cisco/network-device Ansible video coming soon.

## Key claims (dated — Ansible concepts he teaches)

_All 2020-05-07, paraphrased unless quoted._

- Ansible = IT automation: automating systems, cloud, and networking — "all the things."
- It's one of the most popular automation tools, owned by Red Hat, super simple, and you
  don't need to know any programming (though it helps).
- The core value: instead of logging into every server to make the same change, you make
  the change once on a **control station** (master/command center), which pushes it out to
  all managed servers.
- The same benefit applies to network engineers managing many routers/switches (e.g.
  updating an NTP peer or a banner across all devices from one place).
- Ansible is **Python-based**.
- Ansible is **agentless** — its favorite feature: it doesn't require installing an agent
  or client on managed machines; it simply connects over **SSH** and runs commands. This
  matters because you can't install software on Cisco IOS routers.
- Ansible isn't just for Linux and Cisco IOS — it manages Azure, AWS, and Windows machines;
  your entire infrastructure.
- It's **free and open source** (like CentOS); installs on nearly any Linux distro and macOS.
  The paid version is **Ansible Tower** (extra bells and whistles), analogous to Red Hat as
  CentOS's paid enterprise version.
- Ansible uses a **push model** ("it's very pushy") — it pushes configuration/commands to
  devices — versus pull models where devices pull config from the control node.
- **Installation** (CentOS): `yum` update → install the **EPEL release** → install ansible.
  About five minutes.
- The **inventory/host file** (`/etc/ansible/hosts`) lists everything Ansible controls
  (Linux machines, routers, switches). Groups are defined with brackets, e.g. `[linux]`,
  with hosts (IPs) listed below.
- Connection variables are set per group with a `[linux:vars]` section — e.g.
  `ansible_user` and `ansible_password`. (Chuck uses `root` + a password for the lab and
  explicitly flags this as bad practice for production — use a dedicated Ansible account.)
- In `ansible.cfg`, set `host_key_checking = false` for a lab to skip SSH host key setup
  (production keeps host key checking on with SSH keys for security).
- **Ad-hoc commands**: `ansible <group> -m ping` (the `-m` flag = **module**) verifies
  reachability; `ansible <group> -a "<command>"` runs arbitrary shell commands on all hosts
  at once (e.g. `cat /etc/os-release`, `reboot`).
- Ansible **gathers facts** about the machines it controls, which you can use to do things.
- A **playbook** is a **YAML** (`.yml`) file that does the same job as ad-hoc commands but
  more organized. YAML = "YAML Ain't Markup Language," a human-readable data serialization
  language.
- Playbook structure (sports metaphor): a playbook contains **plays**, and each play
  contains **tasks**; each task uses a **module** (small programs that define the desired
  state). Example: use the yum module to ensure Nano is at its latest version.
- Run a playbook with `ansible-playbook <file>.yml`; output shows gathering facts, the task,
  and a **play recap** showing per-node status and what changed.
- **Idempotence** ("idempotency") is what makes Ansible great: running the playbook again
  makes **zero changes** because the desired state already exists — Ansible only makes a
  change if it has to.
- You reverse a change by editing the task state from `latest` to `absent` (removes Nano).
- To go further: read the Ansible module documentation and online examples.
- Automation in IT (networking, sysadmin, cloud) "is here to stay" and is the future;
  managing large networks/infrastructure becomes impossible without it.
