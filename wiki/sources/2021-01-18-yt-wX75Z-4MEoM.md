# you need to learn Virtual Machines RIGHT NOW!! (Kali Linux VM, Ubuntu, Windows)

## Summary

A landmark entry in Chuck Keith's signature "you need to learn X RIGHT NOW" tutorial
series, this time on virtual machines (VMs). Chuck argues that anyone learning hacking,
Linux, or IT in general needs a VM, then walks through the full setup end-to-end on a
Windows laptop: what a VM is, the hypervisor concept, type 1 vs. type 2 hypervisors,
enabling hardware virtualization in the BIOS, downloading OS ISOs (Kali Linux, Ubuntu),
installing Oracle VirtualBox plus its extension pack, creating a VM (memory, virtual
hard disk, CPU allocation), installing Kali Linux as a guest, and exploring VirtualBox
features (pause, save state, reset, clone, snapshots, shared clipboard, drag-and-drop,
NAT vs. bridged networking, shared folders, per-VM VPN). He frames the VM as "a computer
inside a computer" and repeatedly calls virtualization "magic." The video closes by
noting VMs are one option alongside Docker and WSL2, with VMs as the prerequisite concept.

## Key claims (dated — VM concepts)

All claims paraphrased from the 2021-01-18 video unless quoted.

- A virtual machine is "a computer inside a computer" — hardware (CPU, RAM, storage, GPU)
  is virtualized and the guest OS behaves as if it is the only real computer on the machine.
- A hypervisor is just an application you install on your OS, like any other program;
  its job is to carve out hardware resources and create virtual computers.
- **Type 2 hypervisor**: installed on top of an existing host OS (Windows/Linux/Mac);
  it must ask the host OS to share resources. Example: Oracle VirtualBox (free, cross-platform).
- **Type 1 hypervisor**: installed directly onto bare-metal hardware with no underlying
  host OS; it has full ("Godlike") control of all hardware. Example: VMware ESXi, which
  Chuck runs in his own homelab; typically found on enterprise Dell/HP servers.
- Terminology: the underlying OS is the **host OS**; VMs created on it are **guest OSes**.
- Both hypervisor types do the same fundamental thing (carve out hardware to make VMs);
  the difference is whether they must share with a host OS (type 2) or own the hardware (type 1).
- Top three reasons to use a VM: (1) learn hacking safely/isolated — TryHackMe and Hack The
  Box recommend connecting via a VM over VPN; (2) learn different operating systems; (3)
  break stuff — experiment freely, then just delete and rebuild without affecting the host.
- Minimum requirements: any computer bought within the last ~10 years for CPU, and 4 GB RAM
  or more (less will strain the host since resources are shared).
- To run a **64-bit** guest OS you must enable hardware virtualization in the BIOS: Intel
  VMX / "Virtualization Technology" for Intel CPUs, or AMD-V for AMD CPUs. 32-bit guests
  don't require this.
- OS install media comes as an **ISO file** — the modern equivalent of a CD install disc.
- VirtualBox needs a separate **extension pack** for features like USB device passthrough.
- When creating a VM: allocate memory carefully (check Task Manager > Performance for a
  baseline with normal apps open); Chuck gives a Kali VM 2 GB RAM on a 16 GB laptop.
- Virtual hard disk: **VDI** is VirtualBox-specific (fine for VirtualBox-only use); other
  formats aid interoperability with hypervisors like VMware Workstation. **Dynamically
  allocated** storage only grows on the physical disk as needed (Chuck's preference);
  **fixed** size gives slightly better performance.
- CPU allocation has the biggest performance impact — bump cores up (four is "a good solid
  number"), but never allocate more than half your cores, especially with multiple VMs.
- Guided "use entire disk" during install only formats the VM's virtual disk, not the host's
  physical disk — it is safe.
- Isolation is the core security benefit: whatever happens in the guest cannot affect the
  host OS; ideal for connecting to shared hacking networks (TryHackMe / Hack The Box).
- VirtualBox VM operations: **pause** (freezes the VM, frees resources), **reset**
  (like unplug/replug), **save state** (resume exactly where you left off), **power off**
  (like pulling the power cord), **clone** (full copy / backup), and **snapshots** (a
  point-in-time "freeze frame" you can restore to after breaking something).
- Networking modes: default **NAT** puts the VM on its own network (e.g. 10.0.2.x) — it is
  invisible to and cannot see the host's home network, which is safer; **bridged adapter**
  gives the VM a real IP on the home network (more convenient, less isolated). Changing it
  may require a reboot.
- Convenience-vs-security tradeoff: shared clipboard, drag-and-drop, and shared folders all
  reduce isolation; for hacking/learning, keep the VM as isolated as possible.
- A VM can run its own VPN client, so the guest can appear to be in another country (e.g.
  France) while the host is in Dallas, Texas.
- VMs are not the only way to get isolated environments — **Docker** and **WSL2** do it too;
  understanding VMs is a prerequisite before moving into those.
- Multiple guests can run at once: Chuck ends up running Kali Linux, Ubuntu, and the Windows
  host OS simultaneously on one laptop.
