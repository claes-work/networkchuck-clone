# intro to cloud hacking (leaky buckets)

## Summary

An ethical-hacking tutorial that merges Chuck's cloud and security threads: he
teaches "cloud hacking" by targeting AWS S3, the "simple storage service" where
much of the internet's data lives. The core concept is the **leaky bucket** — an
S3 bucket left publicly readable (or exposed to any authenticated AWS user)
through **misconfiguration**. Chuck opens with an explicit ethics disclaimer
(techniques are real and must be used only with explicit permission and within
the rules), then frames the problem: the cloud is prone to misconfigurations,
there is a subculture of "leak hunters," and tools/sites like GrayHat Warfare let
you search public buckets for keywords (e.g. "password" `.txt` files).

He first builds an S3 bucket himself to explain how it "ticks," using it to show
the **defensive lesson**: modern AWS defaults now block all public access and
enable encryption by default, but that was **not** always the case — so old
buckets created "back in the day" remain exposed, and because S3 permission
policies are JSON and genuinely complex, a single typo when granting access can
accidentally leave sensitive data (credentials, technical docs) public.

The hands-on portion walks the **flaws.cloud** CTF (created by Scott Piper) using
the AWS CLI. Level 1: confirm a domain points to S3 via `nslookup` (forward then
reverse DNS), list a public bucket with `aws s3 ls --no-sign-request`, and copy
out a secret file. Level 2: a bucket denied to anonymous users is still readable
by *any* authenticated AWS account (the old "authenticated users" ACL pitfall) —
so a free AWS account plus an IAM user grants access. Level 3: leaked AWS access
keys committed to a `.git` history are recovered via `git checkout` of an old
commit, then used to enumerate other buckets. He closes with **AWSBucketDump**, a
Python CLI tool that mass-searches buckets by wordlist/keywords and can auto-
download contents. The sponsor is ITPro by ACI Learning ("Intro to AWS Pentesting"
by Daniel Lowry).

## Key claims (dated — concept + defense)

All dated 2023-07-10 (per source):

**Concept / offense**
- AWS S3 ("simple storage service") is external-hard-drive-style object storage
  where companies and much of the internet store data; a hacked bucket is like a
  hacked Google Drive or Dropbox. [paraphrase]
- A "leaky bucket" is a misconfigured, publicly exposed S3 bucket; there is a
  hacker subculture of "leak hunters" who search for them. [paraphrase]
- Sites like GrayHat Warfare let anyone search public S3 buckets for keywords and
  file types (e.g. password `.txt` files). [paraphrase]
- Cloud hacking needs almost no tooling — just a computer (Windows/Mac/Linux),
  though Linux (Ubuntu/Kali) plus a free AWS account maximizes what you can do. [paraphrase]
- Confirming a domain is backed by S3: run `nslookup` on the domain for its IP,
  then a reverse `nslookup` on that IP, which reveals the S3 hostname. [paraphrase]
- A public bucket can be listed and copied from with no credentials using
  `aws s3 ls s3://<bucket> --no-sign-request` and `aws s3 cp ... --no-sign-request`. [paraphrase]
- The old "any authenticated AWS user" access setting means *anyone* with any AWS
  credentials — not just your own account — can read the bucket; a free IAM user
  defeats it. [paraphrase]
- Leaked AWS keys committed to a website's `.git` folder can be recovered by
  inspecting logs and doing `git checkout` of the offending commit, then reused to
  enumerate the account's other buckets (`aws s3 ls --profile <p>`). [paraphrase]
- AWSBucketDump is a Python CLI that mass-searches buckets by a name list (`-l`)
  and keyword list (`-g`) and can auto-download contents (fills a disk fast). [paraphrase]

**Defense**
- Modern AWS defaults now block all public access to new buckets and enable
  encryption by default — a good default, but it was not always so; pre-default
  buckets still exist unprotected in the wild. [paraphrase]
- S3 security is hard to configure correctly: policies are JSON code, and when
  trying to allow the right access while blocking everything else, one typo can
  leave a bucket open — this happens constantly. [paraphrase]
- Sensitive material (credentials, technical documents) in a misconfigured bucket
  can cause a real breach; historical examples include US voter records, WWE,
  Verizon, and Time Warner Cable. [paraphrase]
- Never commit access keys to Git; if keys leak, revoke them (don't just try to
  cover the mistake without revoking) — Chuck revokes the demo IAM keys shown
  on screen. [paraphrase]

**Ethics**
- The techniques shown are real and must only be used ethically — meaning explicit
  permission and following all the rules; the targets in the video are sanctioned
  (flaws.cloud invites you to hack it). [paraphrase]
